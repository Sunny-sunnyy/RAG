Core Theory - Lý thuyết cốt lõi

FAQ tool - công cụ FAQ

Concept - Khái niệm: một deterministic tool cho phép agent lấy nguyên văn câu trả lời FAQ khi nhận ra câu hỏi phù hợp.

Methods / Techniques - Phương pháp / kỹ thuật: lưu FAQ dạng JSONL, đưa danh sách câu hỏi vào instructions, tra answer qua tool khi cần.

Tools / Frameworks - Công cụ / framework: Python dict lookup, `@function_tool`, Agents SDK tool schema.

Trade-off - Đánh đổi: grounded hơn nhưng cần curated data và bảo trì riêng ngoài vector memory.

When to use - Khi nào nên dùng: khi có nhóm câu hỏi lặp lại, câu trả lời dài, quan trọng, và muốn ổn định cao.

Hybrid retrieval - truy hồi lai

Concept - Khái niệm: không phụ thuộc một nguồn retrieval duy nhất; agent phối hợp vector memory với structured FAQ lookup.

Methods / Techniques - Phương pháp / kỹ thuật: luôn dùng memory tool, và nếu query match FAQ thì dùng thêm FAQ tool song song.

Tools / Models / Services - Công cụ / mô hình / dịch vụ: Qdrant MCP server, FAQ function tool, GPT-5.5 hoặc model tương đương.

Trade-off - Đánh đổi: độ đúng tăng, nhưng orchestration phức tạp hơn và đòi prompt rõ hơn.

When to use - Khi nào nên dùng: khi domain có cả câu hỏi mở lẫn câu hỏi “known canonical answers”.

Streaming - phản hồi theo luồng

Concept - Khái niệm: thay vì chờ xong toàn bộ answer, UI nhận dần text và trạng thái tool-use.

Methods / Techniques - Phương pháp / kỹ thuật: dùng `Runner.run_streamed`, loop qua `stream_events()`, phân loại event types.

Tools / Frameworks - Công cụ / framework: Agents SDK streamed response, `ResponseTextDeltaEvent`, Gradio generator callback.

Trade-off - Đánh đổi: UX tốt hơn nhưng code phức tạp và “janky” hơn so với blocking call.

When to use - Khi nào nên dùng: khi muốn người dùng thấy hệ đang gọi tool và phản hồi dần như chat app thật.

Instant answer path - nhánh trả lời tức thì

Concept - Khái niệm: nếu input rất có cấu trúc như `Q37`, không cần tốn chi phí LLM; code có thể trả lời trực tiếp.

Methods / Techniques - Phương pháp / kỹ thuật: parse pattern input, map sang question number, gọi local lookup.

Tools / Frameworks - Công cụ / framework: Python string logic, FAQ dict, Gradio chat callback.

Trade-off - Đánh đổi: nhanh và rẻ hơn, nhưng dễ làm lệch session state nếu không sync với conversation memory.

When to use - Khi nào nên dùng: khi có những query cực kỳ deterministic và không cần reasoning.

Key relationship - Quan hệ cốt lõi: vector memory giải quyết câu hỏi mở và context nền; FAQ tool giải quyết câu hỏi canonical; instant-answer path giải quyết câu hỏi cực kỳ deterministic; streaming chỉ là lớp UX để người dùng nhìn thấy hệ thống hoạt động.
Section 6

Workflow / Pipeline - Quy trình / luồng hoạt động

High-level flow - Luồng tổng quát

  1. Đọc local vector DB đã ingest từ bài trước.
  2. Đọc `knowledge/faq.jsonl` và parse từng dòng JSONL.
  3. Xây system instructions với ba phần: Role, Memory, FAQ.
  4. Chỉ đưa danh sách câu hỏi FAQ vào prompt, không đưa toàn bộ câu trả lời.
  5. Tạo `faq_tool` từ Python function để lookup answer theo question number.
  6. Nếu user nhập `Qn`, trả lời trực tiếp bằng code path.
  7. Nếu không, chạy agent với memory tool + FAQ tool, stream answer về Gradio.

Control flow - Luồng điều khiển

Có ba nhánh chính: direct path cho `Qn`, memory-only tendency cho câu hỏi mở không match FAQ rõ ràng, và memory + FAQ parallel path cho câu hỏi vừa cần context vừa cần canonical answer.

Trong transcript, tác giả nhấn mạnh rằng agent có thể gọi cả `qdrant-find` và `faq_tool` song song để tổng hợp câu trả lời tốt hơn. Đây là một bước “agentic hơn” so với bài trước, nơi memory là nguồn chính gần như duy nhất lúc answer.

Pipeline summary - Tóm tắt pipeline: user message -> kiểm tra `Qn` shortcut -> nếu match thì direct FAQ lookup -> nếu không match thì agent -> `qdrant-find` + tùy chọn `faq_tool` -> streamed answer -> Gradio UI.
Section 7

Techniques - Kỹ thuật sử dụng

Prompt slimming - làm gọn prompt

Chỉ liệt kê questions, không nhét answers vào context. Đây là kỹ thuật giảm context bloat nhưng vẫn giữ agent awareness về những FAQ tồn tại.

Tool schema as instruction - schema tool như một phần hướng dẫn

`@function_tool` không chỉ expose function; nó còn tạo description và JSON schema để model biết tool nhận tham số gì và dùng trong trường hợp nào.

Parallel tool use - gọi tool song song

Instructions nói rõ “make all tool calls in parallel”. Ý tưởng là vừa lấy memory nền vừa lookup FAQ cụ thể để tăng answer quality mà không phải serialize mọi bước.

Bypass logic - logic bỏ qua LLM

Đây là kỹ thuật tối ưu cost/latency rất thực tế: nhận ra khi nào bài toán chỉ là lookup chứ không cần language reasoning.

Cluster - Cụm kỹ thuật Problem - Vấn đề Popular choices - Lựa chọn phổ biến Trade-off - Đánh đổi When to use - Khi nào nên dùng

Section 9

Implementation Insights - Góc nhìn triển khai

What the code does - Code đang làm gì

[next_level.ipynb](G:/RAG/ed/Agentic_Rag_MCP/expert-next/next_level.ipynb) bắt đầu bằng cách mở local vector DB, rồi đọc [faq.jsonl](G:/RAG/ed/Agentic_Rag_MCP/expert-next/knowledge/faq.jsonl) thành danh sách FAQ. Sau đó notebook build một `instructions` string theo kiểu markdown: `Role`, `Memory`, `FAQ`, rồi append tất cả FAQ questions theo số thứ tự.

Tiếp theo, code tạo `faqs_lookup`, viết `find_faq(question_number)`, rồi bọc nó thành `faq_tool` bằng `@function_tool`. Agent cuối cùng được tạo với `tools=[faq_tool]` và `mcp_servers=[vectorstore_mcp]`, nghĩa là nó vừa có function tool vừa có MCP memory tool.

Why this design - Vì sao chọn thiết kế này

Thiết kế này giải đúng một vấn đề cụ thể: FAQ answers quá dài để nạp hết vào prompt, nhưng lại quá quan trọng để chỉ giao cho semantic search. Thay vì chọn một trong hai, notebook chọn phương án ở giữa: để model thấy “question catalog” và tự gọi tool để lấy “answer body”.

Đây là một design rất senior theo nghĩa thực dụng: dùng LLM ở nơi cần judgment, dùng deterministic lookup ở nơi đã có ground truth rõ ràng.

Code map - Bản đồ code quan trọng
  • next_level.ipynb: bài chính của buổi này, chứa toàn bộ logic upgrade.
  • ingest.ipynb: dependency từ bài trước để tạo vector DB.
  • knowledge/faq.jsonl: dữ liệu FAQ có cấu trúc, 58 dòng JSONL.
  • FAQ.md: spec để tạo hoặc cập nhật `faq.jsonl`.
  • styles.py: chỉ lo UI/UX cho Gradio.
Important implementation detail - Chi tiết dễ bị bỏ qua
  • Tool description matters: docstring của `faq_tool` chính là phần description agent sẽ nhìn thấy.
  • Schema matters: function tool tự tạo JSON schema cho param `question_number`, giúp model gọi tool đúng format.
  • Prompt budget matters: chỉ đưa questions vào prompt là một quyết định tiết kiệm context window rất quan trọng.
  • State consistency risk: trong repo `expert-next` hiện tại, nhánh `instant answer` chỉ `yield` kết quả trực tiếp mà không ghi vào `SQLiteSession`; điều này đơn giản hơn nhưng có thể làm follow-up context kém nhất quán hơn so với bản có sync session.
Section 10

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - Trường hợp dễ fail

  • FAQ mismatch: model không nhận ra câu hỏi người dùng map vào FAQ nào, nên chỉ dùng vector memory và trả lời kém chắc.
  • Over-reliance on memory: instructions không đủ rõ làm agent không tận dụng FAQ tool khi đáng ra nên dùng.
  • Prompt overload risk: nếu đưa cả questions lẫn answers vào prompt, context sẽ phình và đắt hơn nhiều.
  • Streaming complexity: event handling dễ rối và khó debug hơn flow blocking đơn giản.
  • Session drift on bypass path: direct answer nhanh nhưng có thể làm conversation history không đồng bộ với hành vi user thấy trên UI.

Common misunderstandings - Hiểu lầm phổ biến

  • “FAQ tool thay thế vector store”: sai; nó bổ sung cho vector memory, không thay thế câu hỏi mở.
  • “Tool càng nhiều càng tốt”: sai; mỗi tool thêm vào làm tăng search space cho agent.
  • “Streaming làm model thông minh hơn”: sai; streaming chỉ đổi cách trình bày output.
  • “Nếu có shortcut thì bỏ luôn agent”: sai; shortcut chỉ giải quyết một lớp query rất hẹp.

Flashcards - Thẻ ôn nhanh

  • Q: Tại sao chỉ đưa FAQ questions vào prompt?
    A: Để model biết câu hỏi nào tồn tại mà không tốn context window cho toàn bộ answers.
  • Q: `@function_tool` làm gì?
    A: Nó biến Python function thành tool có description và parameter schema để agent gọi được.
  • Q: Hybrid retrieval trong bài này là gì?
    A: Kết hợp vector memory retrieval với FAQ lookup có cấu trúc.
  • Q: `Q37` shortcut cho thấy điều gì?
    A: Không phải bài toán nào cũng cần LLM; đôi khi direct lookup là giải pháp tốt nhất.
  • Q: Streaming thêm gì cho người dùng?
    A: Cảm giác realtime và khả năng thấy tool-status khi agent đang làm việc.

Interview Q&A - Hỏi đáp phỏng vấn

  • Q: Hãy giải thích vì sao FAQ tool làm hệ “agentic hơn”.
    A: Vì agent không còn chỉ có một kiểu lấy thông tin là semantic recall từ vector DB. Nó được trao thêm một capability có cấu trúc, và phải tự quyết định khi nào nên gọi capability đó để lấy ground-truth answer. Agentic hơn ở đây là tăng quyền điều phối nguồn tri thức, không phải chỉ tăng số dòng prompt.
  • Q: Nếu bạn muốn productionize pattern này, bạn sẽ cải tiến gì?
    A: Tôi sẽ thêm telemetry cho tool usage, đánh giá tách riêng FAQ-hit accuracy và memory-only accuracy, đồng bộ direct-answer path với session state, và cân nhắc thêm classifier/routing layer nếu số loại tool tăng lên quá nhiều.

Quick Memory - Ôn nhanh cuối bài

Must remember - Phải nhớ

  2. Không nên nhét tất cả canonical answers vào prompt nếu có thể tra chúng bằng tool.

  4. Streaming là UX improvement, không phải retrieval improvement.
  5. Bypass LLM là một design choice đúng khi query cực kỳ deterministic.
