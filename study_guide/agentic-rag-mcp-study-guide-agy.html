Section 1

Cheat Sheet - Tóm tắt một trang

Core idea - Ý tưởng cốt lõi

**Agentic RAG với MCP** thay thế luồng RAG truyền thống cứng nhắc bằng việc trao quyền cho LLM tự quyết định khi nào cần gọi công cụ. Sử dụng giao thức mở **MCP** giúp tái sử dụng các server cào dữ liệu và lưu trữ vector đã được phát triển sẵn, giúp giảm thiểu tối đa mã nguồn tự viết và dễ dàng kết nối tác tử với mọi môi trường hệ thống.

Must know - Cần nhớ

  • MCP (Model Context Protocol) hoạt động thông qua giao tiếp Standard I/O (stdio), cho phép LLM tự do gọi các công cụ ngoài.
  • Short-term memory - Bộ nhớ ngắn hạn được lưu trữ bởi SQLiteSession, duy trì lịch sử chat.
  • Long-term memory - Bộ nhớ dài hạn lưu trong Qdrant Vector DB thông qua máy chủ MCP.
  • Instant Answer Routing rẽ nhánh nhanh câu hỏi dạng mã tĩnh (như "q5") để lấy ngay từ FAQ, không tốn API token của LLM.

Góc phỏng vấn - Interview angle

Khi được hỏi về khả năng mở rộng của hệ thống Agentic RAG, hãy giải thích cách **MCP** tách biệt tầng chạy mô hình (LLM Client) và tầng cung cấp công cụ (MCP Servers). Tác tử có thể gọi song song (parallel tool calls) các máy chủ MCP fetch (cào dữ liệu) và Qdrant (lưu trữ vector) để thực hiện các tác vụ phức tạp một cách tự động và linh hoạt.

Advanced RAG Pro (Day 5 Pro) Tối ưu hóa khâu truy xuất: Viết lại query ([rewrite_query](file:///G:/RAG/pro_implementation/answer.py#L90)), mở rộng query (expansion), xếp hạng lại kết quả bằng mô hình [rerank](file:///G:/RAG/pro_implementation/answer.py#L54). Tăng độ chính xác (accuracy) và nDCG/MRR của tài liệu tìm thấy. Tốn kém chi phí token, latency tăng do phải gọi LLM Reranker.
3. Agentic RAG (InsureLLM) Tự động định tuyến (Intent-aware Routing), chia nhỏ tài liệu phân cấp (Parent-Child chunking) và tự đánh giá chất lượng câu trả lời để retry ([evaluate_answer_quality](file:///G:/RAG/Agentic-RAG-InsureLLM/implementation/answer.py#L501)). 4. Agentic RAG với MCP (Buổi hiện tại) Tác tử tự điều khiển việc nạp và truy xuất dữ liệu thông qua giao thức tiêu chuẩn **MCP** (Fetch, Qdrant). Tự tạo công cụ bằng code Python và tối ưu luồng tĩnh. Không còn code slop; cực kỳ dễ mở rộng thêm công cụ (SQL, Gmail, Slack) mà không cần viết lại ứng dụng. Cần cấu hình môi trường chạy máy chủ MCP (như node/wsl), dễ bị nghẽn ở lớp giao tiếp stdio.
Section 5

Core Theory - Lý thuyết cốt lõi

Khái niệm cốt lõi - Key concepts

  • Model Context Protocol (MCP) - Giao thức ngữ cảnh mô hình: Là một giao thức mở (open protocol) chuẩn hóa cách thức kết nối LLM với môi trường bên ngoài (cơ sở dữ liệu, file system, APIs).
  • Short-term memory - Bộ nhớ ngắn hạn: Cơ chế ghi nhớ ngữ cảnh lịch sử trò chuyện trong phiên làm việc hiện tại, giúp duy trì mạch hội thoại mạch lạc.
  • Long-term memory - Bộ nhớ dài hạn: Kho lưu trữ vĩnh viễn các tri thức của tác tử (trong cơ sở dữ liệu vector), được truy xuất thông qua phép so khớp tương đồng ngữ nghĩa.

Quan hệ giữa các thành phần - Component Relationships

Tác tử đóng vai trò bộ não điều phối trung tâm. Khi nhận yêu cầu từ người dùng: Tác tử sẽ ghi câu hỏi vào **Short-term memory** (SQLite). Đồng thời, tác tử sử dụng **MCP** để truy vấn **Long-term memory** (Qdrant Vector DB). Nhờ vào mô tả công cụ rõ ràng (JSON Schema), tác tử tự quyết định gọi công cụ nào phù hợp để kéo context, sau đó tổng hợp câu trả lời cuối cùng cho người dùng.

Section 6

Workflow / Pipeline - Quy trình / luồng hoạt động

1. Luồng nạp tri thức - Ingestion Pipeline

  1. Khởi chạy tác tử `Ingester` liên kết với Fetch MCP Server và Qdrant MCP Server.
  2. Tác tử nhận lệnh thu thập tri thức từ một URL cho trước.
  3. Gọi công cụ `fetch` để lấy nội dung trang web thô và làm sạch text.
  4. Gọi công cụ `qdrant_find` để kiểm tra thông tin có bị lặp với tri thức cũ trong Vector DB hay không.
  5. Nếu chưa tồn tại, tác tử trích xuất tri thức chính (dưới 1-2 câu gọn gàng) và gọi công cụ `qdrant_store` để ghi vào bộ nhớ dài hạn.
  6. Lặp lại cho đến khi tích lũy đủ 10 unique memories cho mỗi trang web.

2. Luồng phục vụ hội thoại - Serving Chat Pipeline

  1. Người dùng gửi câu hỏi từ Gradio UI chat.
  2. Kiểm tra xem câu hỏi có thuộc dạng nhập tắt mã FAQ tĩnh (ví dụ: "q2"):
    • Nếu đúng: Lấy thẳng câu trả lời từ FAQ và lưu lại lịch sử chat vào SQLiteSession (Không gọi LLM).
    • Nếu sai: Tiến hành bước 3.
  3. Kích hoạt tác tử `Expert` được liên kết với Qdrant MCP Server và custom [faq_tool](file:///G:/RAG/ed/Agentic_Rag_MCP/expert-main/next_level.ipynb#L173).
  4. Tác tử chạy vòng lặp suy luận, tự gọi công cụ tìm kiếm vector (`qdrant_find`) hoặc tra FAQ tĩnh (`faq_tool`) để thu thập ngữ cảnh.
  5. Tác tử tổng hợp ngữ cảnh, sinh câu trả lời và phát trực tiếp (stream) kết quả về UI chat.
Section 7

Techniques - Kỹ thuật sử dụng

Custom Python Function Tool

Sử dụng decorator `@function_tool` của OpenAI Agents SDK để tự động chuyển một hàm python thường thành công cụ của tác tử. SDK tự động trích xuất type hints và docstrings của hàm để sinh mô tả JSON Schema cung cấp cho mô hình gọi công cụ.

Instant Answer Routing

Rẽ nhánh xử lý nhanh cho các truy vấn tĩnh (static queries) có độ ưu tiên cao. Nếu người dùng nhập mã câu hỏi cố định, hệ thống trả về kết quả ngay lập tức để tiết kiệm token và thời gian phản hồi của mô hình.

Streaming Event Processing

Đọc luồng sự kiện từ `Runner.run_streamed` và trích xuất thông tin delta. Giúp chatbot hiển thị trạng thái động cho người dùng (ví dụ: "Calling faq_tool..." hoặc "Tool returned. Thinking...") trước khi xuất kết quả dạng text.

Section 10

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - Các lỗi thường gặp

  • Lỗi kết nối stdio trên Windows (WSL2): Máy chủ MCP chạy trên Windows đôi khi không tương tác được với stdio của môi trường WSL2. Khắc phục bằng cách điều chỉnh cấu hình `.wslconfig` với `networkingMode=mirrored` và `dnsTunneling=true` để thông suốt kết nối mạng của máy ảo.
  • Trùng lặp tri thức trong Vector DB: Nếu tác tử chạy nạp dữ liệu nhiều lần mà không kiểm tra trùng (deduplication check), Qdrant sẽ lưu trữ nhiều bản ghi trùng nghĩa, gây loãng context tìm kiếm. Khắc phục bằng cách yêu cầu tác tử luôn gọi `qdrant_find` kiểm tra trước khi ghi.
  • Tool Calling Loop: LLM liên tục gọi đi gọi lại công cụ tìm kiếm vector với các từ khóa giống nhau khi không tìm thấy kết quả phù hợp. Khắc phục bằng cách cấu hình prompt nghiêm ngặt giới hạn số lần gọi tool hoặc cài đặt giới hạn vòng chạy `max_turns` ở mức an toàn (ví dụ: 10-20).

Hiểu lầm phổ biến - Common misunderstandings

  • "MCP chỉ chạy được thông qua máy chủ Internet": Sai. MCP hoạt động thông suốt qua giao tiếp dòng lệnh tiêu chuẩn (stdio) của hệ thống cục bộ tại local.
  • "Luôn cần gọi LLM để có phản hồi thông minh": Sai. Các câu hỏi tĩnh chuẩn hóa nên được lọc và trả kết quả tức thì (Instant Answer) để tiết kiệm tài nguyên và tăng độ ổn định của hệ thống.
  • "Vector Database tự động lọc trùng lặp dữ liệu đầu vào": Sai. Hầu hết các Vector DB lưu trữ bản ghi phẳng theo ID tăng dần; lập trình viên hoặc tác tử phải tự kiểm tra tương đồng ngữ nghĩa trước khi thêm bản ghi mới.

Interview Takeaways - Ý chính cho phỏng vấn

Các câu hỏi trọng tâm - Core Questions

  • Giải thích cách giao thức MCP hoạt động và tại sao nó tối ưu hơn việc tự viết mã tích hợp SDK của bên thứ ba?
  • Làm thế nào để kết hợp hiệu quả giữa bộ nhớ ngắn hạn (Short-term memory) và bộ nhớ dài hạn (Long-term memory) trong ứng dụng Agentic RAG?
  • Cơ chế Instant Answer giúp ích gì trong việc thiết kế chatbot AI quy mô doanh nghiệp?

Khung câu trả lời mẫu - Answer Frame

Khi trả lời, hãy đi từ **Mục tiêu hệ thống (Goal)** -> **Cách hoạt động (Mechanism)** -> **Đánh đổi (Trade-off)** -> **Cách khắc phục (Mitigation)**. Ví dụ: "MCP giúp chuẩn hóa việc AI sử dụng công cụ bằng stdio. Điểm đánh đổi là tăng độ trễ và có thể gặp lỗi kết nối stdio trên môi trường Windows. Chúng tôi khắc phục bằng cách sử dụng các thư viện quản lý kết nối an toàn (MCPServerStdio) và cơ chế retry tự động (Tenacity)."

Section 12

Interview Q&A / Flashcards - Câu hỏi ôn tập

Flashcards - Thẻ ôn nhanh

  • Q: Giao thức mặc định của Qdrant MCP server dùng mô hình nhúng nào?
    A: all-MiniLM-L6-v2 từ sentence-transformers (nhẹ và cực kỳ nhanh).
  • Q: Làm sao để tác tử OpenAI SDK nhận diện tham số của một Python function tool?
    A: SDK tự động phân tích type hints và nội dung docstring của hàm được bọc bởi `@function_tool`.
  • Q: Tại sao cần sử dụng SQLiteSession cho lịch sử chat thay vì lưu vào RAM?
    A: Lưu vào SQLite giúp bền vững dữ liệu khi khởi động lại chatbot và dễ quản lý phiên làm việc đa người dùng.

Hỏi đáp chuyên sâu - Interview Q&A

  • Q: Nếu MCP Server bị timeout hoặc ngắt kết nối stdio giữa chừng, làm sao ứng dụng của bạn tự phục hồi?
    A: Trong mã nguồn, tôi bọc các hàm khởi tạo tác tử và chạy dòng lệnh trong Tenacity decorator `@retry(wait=wait)` để tự động phục hồi kết nối. Đồng thời, thiết lập thời gian timeout hợp lý (ví dụ: 120 giây) trong `MCPServerStdio` để giải phóng tài nguyên bị treo.
  • Q: Cách tối ưu chi phí token khi người dùng chat liên tục với tác tử RAG có kết nối Vector DB?
    A: Tôi triển khai cơ chế rẽ nhánh phản hồi tức thì (Instant Answer Routing) cho các mã tin nhắn tĩnh. Đồng thời sử dụng mô hình LLM siêu tốc, giá rẻ (như các mô hình chạy qua Cerebras) cho các tác vụ định tuyến hoặc hỏi đáp đơn giản.
Section 13

Quick Memory - Ôn nhanh cuối bài

Quy tắc cốt lõi - Key Rules

  1. MCP Server chạy độc lập bên ngoài ứng dụng AI, giao tiếp qua Standard I/O (stdio).
  2. Mô tả công cụ (tools schema) càng rõ ràng thì LLM gọi công cụ càng chính xác.
  3. Sử dụng `@function_tool` để viết nhanh các công cụ python cục bộ.
  4. Dùng `Runner.run_streamed` để cải thiện trải nghiệm người dùng cuối bằng cách stream nội dung trả về.

Tự kiểm tra - Self-check

  • Bạn đã cấu hình đúng cổng mạng và đường dẫn thư mục Vector DB cục bộ trong biến môi trường chưa?
  • Hàm định tuyến Instant Answer có khả năng ghi lịch sử chat vào SQLite trước khi trả kết quả không?
  • Custom tool của bạn đã được viết docstring mô tả chi tiết các tham số đầu vào và đầu ra chưa?
