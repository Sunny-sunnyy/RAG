Section 1

Cheat Sheet - Tóm tắt một trang

Core idea - Ý chính

Agentic RAG ở phiên bản đơn giản nhất chỉ là: LLM + retrieval tool. Khi model tự quyết định lúc nào cần gọi semantic search để lấy context phục vụ answer, RAG đã chuyển từ pipeline cứng sang hệ thống có decision layer.

Must know - Cần nhớ

  • Traditional RAG: retrieve trước, rồi mới gọi LLM.
  • Agentic RAG: LLM có tool retrieval và tự gọi tool.
  • Retriever agent: tách trách nhiệm retrieval khỏi answer.
  • Research agent: một retriever nhiều tool như vector search, SQL, web search, memory.

Interview angle - Góc phỏng vấn

Nếu bị hỏi về Agentic RAG, cần nói được: mục tiêu, cơ chế tool calling, trade-off với RAG thường, và vì sao evaluation mới quyết định tool nào đáng dùng.

One-line summary - Tóm tắt 1 dòng: Agentic RAG không nhất thiết phức tạp; nó trở nên mạnh hơn khi retrieval được đặt vào tay một agent có tool và được tối ưu bằng thực nghiệm.

Section 5

Core Theory - Lý thuyết cốt lõi

Semantic retrieval - truy xuất theo ngữ nghĩa

Concept - Khái niệm: dùng embedding model - mô hình nhúng vector để biến text và query thành vector, rồi tìm những điểm gần nhau về nghĩa trong vector space.

Methods / Techniques - Phương pháp / kỹ thuật: embeddings, nearest-neighbor search, vector store query.

Tools / Models / Frameworks / Services - Công cụ / mô hình / framework / dịch vụ: OpenAI embeddings, Chroma, t-SNE cho visualization.

Trade-off - Đánh đổi: mạnh hơn keyword search nhưng cần preprocessing, vector storage và tuning top-k.

When to use - Khi nào nên dùng: khi câu hỏi và tài liệu có thể diễn đạt khác từ nhưng cùng nghĩa.

Traditional RAG - RAG truyền thống

Concept - Khái niệm: hệ thống retrieve relevant context trước, sau đó nhét context vào prompt rồi mới gọi LLM trả lời.

Methods / Techniques - Phương pháp / kỹ thuật: top-k semantic retrieval, prompt augmentation.

Tools / Models / Frameworks / Services - Công cụ / mô hình / framework / dịch vụ: Chroma, chat completion API.

Trade-off - Đánh đổi: đơn giản, dễ hiểu, dễ debug, nhưng orchestration cứng và không linh hoạt theo từng query.

When to use - Khi nào nên dùng: MVP, demo nhỏ, hoặc use case có retrieval flow tương đối đồng nhất.

Agentic RAG - RAG có tác tử

Concept - Khái niệm: LLM được cấp tool retrieval và tự quyết định khi nào cần gọi tool để lấy context.

Methods / Techniques - Phương pháp / kỹ thuật: tool calling, JSON schema generation từ type hints và docstrings, agent run loop.

Tools / Models / Frameworks / Services - Công cụ / mô hình / framework / dịch vụ: OpenAI Agents SDK, @function_tool.

Trade-off - Đánh đổi: linh hoạt hơn nhưng khó kiểm soát hơn; agent có thể gọi tool không tối ưu hoặc trả lời khi đáng ra nên chỉ retrieve.

When to use - Khi nào nên dùng: khi retrieval không còn là bước luôn luôn phải chạy trước theo một pattern cứng.

Retrieval / Research agent - tác tử truy xuất / nghiên cứu

Concept - Khái niệm: tách retrieval thành một agent riêng để tối ưu cho nhiệm vụ tìm context, không trực tiếp answer.

Methods / Techniques - Phương pháp / kỹ thuật: prompt specialization, agent-as-tool, multi-tool orchestration.

Tools / Models / Frameworks / Services - Công cụ / mô hình / framework / dịch vụ: vector DB, SQL query, web search, graph DB, memory stores.

Trade-off - Đánh đổi: rõ trách nhiệm hơn nhưng tăng latency, complexity, cost và surface area để debug.

When to use - Khi nào nên dùng: khi context đến từ nhiều nguồn hoặc cần nhiều kiểu truy vấn khác nhau.

Section 6

Workflow / Pipeline - Quy trình / luồng hoạt động

1

Build knowledge base - Xây knowledge base

Notebook 1 đọc facts.jsonl, tạo embeddings cho 36 facts về Ed, rồi lưu text + vectors vào Chroma vector store - kho lưu trữ vector.

2

Traditional query flow - Luồng query truyền thống

User hỏi -> embed query -> query Chroma lấy top-3 docs -> ghép context vào prompt -> gọi chat model -> trả lời. Không có decision layer trong LLM.

3

Basic agentic flow - Luồng agentic cơ bản

User hỏi -> expert agent nhận question -> agent tự gọi retrieve_relevant_context_tool nếu thấy cần -> lấy context -> trả answer. Retrieval giờ là tool, không còn là bước cứng ở ngoài.

4

Retriever delegation - Ủy quyền cho retriever

Expert agent không gọi semantic search trực tiếp nữa, mà gọi một retrieval agent - tác tử truy xuất. Retrieval agent này có instructions riêng và có thể được biến thành một tool cho expert agent.

5

Research agent extension - Mở rộng thành research agent

Retrieval agent được trang bị thêm SQL, web search, graph DB, memory. Khi đó Agentic RAG trở thành một phần của hệ kiến trúc agent lớn hơn, nơi retrieval là một nhiệm vụ nghiên cứu đa nguồn.

Section 7

Techniques - Kỹ thuật sử dụng

Embedding-based search - tìm kiếm dựa trên vector nhúng

Đây là kỹ thuật nền để RAG hoạt động. Query và documents được đưa vào cùng một không gian vector, rồi dùng độ gần để chọn context liên quan nhất.

Tool wrapping - đóng gói function thành tool

Hàm retrieval được bọc bằng @function_tool, thêm type hints và docstring để SDK tự sinh description và JSON schema cho model dùng.

Agent-as-tool - agent như một công cụ

Retrieval agent được biến thành tool bằng as_tool(). Đây là pattern quan trọng để ghép nhiều tác tử mà vẫn giữ giao diện gọi đơn giản.

Optimization insight - Insight tối ưu: điểm mới của buổi học không phải một retrieval algorithm mới, mà là thay đổi chỗ đặt quyết định: orchestration được đẩy từ code cứng sang agent.

Section 10

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - Trường hợp dễ fail

  • Agent answers too early: model trả lời luôn thay vì retrieve, đặc biệt nếu instructions không đủ chặt.
  • Tool misuse - dùng tool không tối ưu: agent gọi tool sai lúc, sai query, hoặc gọi quá nhiều lần gây tốn cost.
  • Research sprawl - research lan man: thêm quá nhiều tool mà không có evaluation, làm system rộng ra nhưng không tốt hơn.
  • False confidence: demo nhỏ chạy tốt không có nghĩa pipeline đó đã sẵn sàng cho dữ liệu thật nhiều nguồn.

Common misunderstandings - Hiểu lầm phổ biến

  • “Agentic RAG phải nhiều agent”: sai; bản tối thiểu chỉ cần LLM có retrieval tool là đã đủ agentic.
  • “Có thêm tool là luôn tốt hơn”: sai; mỗi tool thêm vào đều tăng latency, cost và complexity.
  • “Semantic search là tìm vector chứ không phải text”: sai; text vẫn là phần giá trị được lưu, vector chỉ là chỉ mục để truy xuất nhanh theo nghĩa.
  • “Research agent là production-ready by default”: sai; nó chỉ là pattern kiến trúc, còn chọn tool nào vẫn phải qua thực nghiệm.

Section 12

Interview Q&A / Flashcards - Câu hỏi phỏng vấn / thẻ ôn tập

Flashcards - Thẻ ôn nhanh

  • Q: Traditional RAG là gì?
    A: Retrieve relevant context trước, augment prompt, rồi mới gọi LLM trả lời.
  • Q: Agentic RAG tối thiểu cần gì?
    A: Một agent có retrieval tool và quyền tự gọi tool đó.
  • Q: Retriever agent khác retrieval tool thế nào?
    A: Tool là function capability; agent là một layer quyết định có thể dùng nhiều tool và có instructions riêng.
  • Q: Research agent dùng khi nào?
    A: Khi context phải được gom từ nhiều nguồn như vector DB, SQL, web search hoặc memory.

Interview Q&A - Hỏi đáp phỏng vấn

  • Q: Vì sao Agentic RAG không chỉ là một buzzword?
    A: Vì nó thay đổi chỗ đặt quyết định trong hệ thống. Retrieval không còn là bước code cứng ở ngoài, mà trở thành một capability agent có thể gọi tự chủ để đạt goal answer question tốt hơn.
  • Q: Nếu thêm SQL và web search vào retriever, bạn kiểm chứng giá trị thế nào?
    A: Tạo test set đại diện, chạy baseline với vector-only retrieval, rồi A/B test các cấu hình tool khác nhau để đo accuracy, latency, cost và failure category. Tool chỉ đáng giữ nếu metric tốt hơn rõ ràng trên use case thật.
