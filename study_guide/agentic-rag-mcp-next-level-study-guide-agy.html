Cheat Sheet - Tóm tắt một trang

Core idea - Ý chính

Khắc phục nhược điểm của RAG truyền thống bằng mô hình kết hợp lai: tìm kiếm ngữ nghĩa mờ trên Vector DB (Qdrant) kết hợp tra cứu dữ liệu tĩnh chính xác (FAQ Tool). Áp dụng bộ định tuyến rẽ nhánh tức thì (Instant Bypass) để không gọi LLM đối với các câu hỏi tĩnh định trước, tối ưu hóa triệt để tài nguyên và tốc độ phản hồi.

Must know - Cần nhớ

  • FAQ Indexing: Chỉ nạp danh sách tiêu đề câu hỏi được đánh số vào Prompt ngữ cảnh của Agent. Tránh phình Prompt.
  • Custom Python Tool: Viết hàm Python tra cứu và bọc bằng `@function_tool` để Agent tự động gọi qua mô tả JSON Schema.
  • Parallel Tool Calling: Cho phép Agent gọi song song Qdrant Vector search và FAQ tool nhằm tối ưu hóa thông tin context.
  • LLM Bypass Routing: Lọc nhanh đầu vào. Nếu là mã câu hỏi tĩnh (như "Q3"), trả kết quả trực tiếp bằng code Python thuần.

Góc phỏng vấn - Interview angle

Nếu bị hỏi về cách thiết kế hệ thống RAG tối ưu cho hàng ngàn câu hỏi thường gặp, hãy trình bày giải pháp: Bypass các câu hỏi tĩnh bằng bộ lọc Regex/Dictionary (Fast Path) để đạt độ trễ ~0ms; đối với các câu hỏi phức tạp hơn (Smart Path), sử dụng Agent điều phối gọi công cụ động thay vì nhồi nhét toàn bộ tri thức tĩnh vào Prompt để tiết kiệm chi phí token và tránh hallucination.

Session Overview - Tổng quan buổi học

Mã nguồn của buổi học [expert-next](file:///G:/RAG/ed/Agentic_Rag_MCP/expert-next) là một mô hình thực tế cực kỳ đắt giá giúp nâng cấp chatbot AI từ trạng thái chỉ tìm kiếm ngữ nghĩa mờ (Vector database) sang một hệ chuyên gia có độ tin cậy tuyệt đối. Hệ thống giải quyết triệt để 3 vấn đề lớn của RAG truyền thống:

1. Độ chính xác thông tin tĩnh

2. Tiết kiệm chi phí vận hành

Cơ chế lọc và bỏ qua LLM (Bypass) giúp giảm tải số lượng API request gửi lên mô hình. Hệ thống chỉ kích hoạt LLM khi người dùng đặt các câu hỏi mở phức tạp.

3. Trải nghiệm người dùng (UX)

Kết hợp streaming hội thoại thời gian thực qua `run_streamed` và cơ chế Client-side Auto-focus giúp người dùng tương tác mượt mà, không bị gián đoạn khi bot đang thực thi các cuộc gọi công cụ (tool calls) ở chế độ chạy nền.

Section 5

Core Theory - Lý thuyết cốt lõi

Các khái niệm then chốt - Key concepts

  • Hybrid Retrieval - Truy xuất lai: Sự phối hợp giữa tìm kiếm không cấu trúc tương đồng (Vector Search) và tìm kiếm có cấu trúc chính xác (FAQ Key Lookup) để tận dụng ưu điểm của cả hai thế giới.
  • Prompt Compression / Context Optimization - Tối ưu hóa ngữ cảnh: Kỹ thuật ẩn thông tin chi tiết vào các điểm cuối công cụ (Tool endpoints) và chỉ tiết lộ danh sách mục lục (Index list) cho LLM để giảm thiểu độ dài Prompt đầu vào.
  • Rule-Based Routing - Định tuyến dựa trên quy tắc: Sử dụng các điều kiện logic lập trình cứng (như regex, regex khớp chuỗi) để xử lý các truy vấn đơn giản, bảo vệ LLM khỏi việc lãng phí tài nguyên tính toán.

Mối quan hệ hoạt động - Component relationships

Khi nhận câu hỏi từ người dùng, hệ thống sẽ rẽ nhánh: Nếu câu hỏi khớp mẫu viết tắt của FAQ, ứng dụng sẽ chạy đường tắt (Bypass LLM), truy xuất trực tiếp từ file [faq.jsonl](file:///G:/RAG/ed/Agentic_Rag_MCP/expert-next/knowledge/faq.jsonl), phản hồi ngay và cập nhật lịch sử vào SQLite (Short-term memory). Nếu câu hỏi là tự do, tác tử được đánh thức. Dựa vào danh sách chỉ mục câu hỏi FAQ được đính kèm trong System Instructions, Agent sẽ quyết định gọi song song `qdrant-find` (tìm dữ liệu mờ) và [faq_tool](file:///G:/RAG/ed/Agentic_Rag_MCP/expert-next/next_level.ipynb#L178) (tìm câu trả lời chính xác). Kết quả của các tool được LLM tổng hợp để tạo câu trả lời mạch lạc nhất.

Cụm kỹ thuật - Technical Cluster Concept - Khái niệm Why it matters - Vì sao quan trọng
Model Context Protocol (MCP) Giao thức chuẩn hóa giúp LLM giao tiếp trực tiếp với môi trường ngoài (file, database) qua Standard I/O. Loại bỏ mã nguồn trung gian (glue code), giúp tái sử dụng các MCP server tiêu chuẩn có sẵn của cộng đồng.

@function_tool decorator Decorator bọc hàm Python thường thành một công cụ có mô tả schema tự động cho LLM Agent. Tự động hóa việc sinh cấu trúc JSON Schema từ Type hints và Docstrings của hàm, giảm thiểu lỗi định nghĩa.

Workflow / Pipeline - Quy trình / luồng hoạt động

Quy trình Ingestion & Serving - Workflow

  1. Pha Ingestion (Nạp tri thức):
    • Tác tử sử dụng `mcp-server-fetch` để tải mã HTML thô từ website.
    • Tác tử trích xuất thông tin, kiểm tra trùng lặp thông qua `mcp-server-qdrant`.
    • Lưu trữ các facts duy nhất vào Qdrant Vector Store dưới dạng Embeddings.
  2. Pha Serving (Phục vụ chat):
    • Gradio nhận yêu cầu chat của người dùng.
    • Gọi hàm [chat](file:///G:/RAG/ed/Agentic_Rag_MCP/expert-next/next_level.ipynb#L280) để phân tích đầu vào.
    • Nếu khớp Regex "Q{number}" -> Trả ngay câu trả lời từ file FAQ, ghi nhận vào bộ nhớ SQLite, bypass LLM.
    • Nếu không khớp -> Khởi tạo Agent với instructions chứa danh sách tiêu đề FAQ.
    • LLM gọi song song `qdrant-find` và [faq_tool](file:///G:/RAG/ed/Agentic_Rag_MCP/expert-next/next_level.ipynb#L178).
    • Stream kết quả sự kiện trả về giao diện Gradio thông qua hàm xử lý sự kiện.

Luồng điều khiển dữ liệu - Control / Data Flow

Mọi tương tác trong pha Smart Path đều được truyền tải dưới dạng các Sự kiện (Events) của OpenAI Agents SDK. Dữ liệu đi qua hàm [text_from_event](file:///G:/RAG/ed/Agentic_Rag_MCP/expert-next/next_level.ipynb#L244) để lọc ra các sự kiện chính:

  • ResponseTextDeltaEvent: Nhận các đoạn text phản hồi được sinh ra từ LLM để nối tiếp tục và hiển thị cho người dùng.
  • tool_called: Phát hiện khi Agent bắt đầu kích hoạt một công cụ. Trả về đoạn mã HTML hiển thị trạng thái đang tải: `Calling {tool_name}...`.
  • tool_output: Phát hiện khi công cụ trả kết quả về cho LLM. Hiển thị trạng thái: `Tool returned. Thinking...`.

2. Instant Answer Regex Matching

3. DOM MutationObserver

Nhúng mã Javascript tùy biến trong [styles.py](file:///G:/RAG/ed/Agentic_Rag_MCP/expert-next/styles.py) để giám sát DOM của trang Gradio. Mã sẽ theo dõi sự thay đổi thuộc tính `disabled` hoặc `readonly` của thẻ `textarea` đầu vào để tự động đặt lại tiêu điểm (focus) ngay khi Bot dừng phản hồi.

Lý do của thiết kế này - Design decisions

- **Sử dụng `@function_tool`**: Giúp lập trình viên tập trung viết logic xử lý bằng Python thuần, giảm thiểu tối đa các khai báo JSON Schema thủ công vốn dễ sai sót. - **Custom CSS/JS**: Khắc phục nhược điểm giao diện mặc định thô sơ của Gradio. Custom CSS giúp UI phẳng, hiện đại và tập trung. Đoạn JS bổ sung giải quyết triệt để vấn đề trải nghiệm người dùng khó chịu khi phải bấm chuột lại vào ô nhập liệu sau mỗi lượt chat.

Phân tích sâu cơ chế gọi công cụ song song - Parallel Tool Calling Deep Dive

Trong OpenAI Agents SDK, khi Agent nhận được Prompt của người dùng (ví dụ: "What job can I get after your courses?"), nó phân tích và thấy rằng câu hỏi này vừa liên quan đến một câu trong danh sách FAQ chỉ mục (câu hỏi số 2), vừa có thể có thêm thông tin chi tiết trong Vector store.

Thay vì thực hiện tuần tự (Sequential) gây tăng độ trễ: gọi tool 1 -> chờ kết quả -> gọi tool 2 -> chờ kết quả; OpenAI Models hỗ trợ gọi công cụ song song (Parallel Tool Calling). LLM sẽ trả về lệnh gọi cả hai công cụ `faq_tool(question_number=2)` và `qdrant-find(query="job opportunities after courses")` trong cùng một lượt phản hồi. SDK sẽ thực thi đồng thời hai hàm này, sau đó gửi toàn bộ kết quả về cho LLM tổng hợp. Điều này giúp tối ưu hóa đáng kể độ trễ hội thoại (conversational latency).

Section 10

Pitfalls - Lỗi / bẫy thường gặp

Các lỗi logic thực tế - Failure modes

  • Mất lịch sử chat ngắn hạn khi Bypass LLM: Trong mã nguồn gốc của [next_level.ipynb](file:///G:/RAG/ed/Agentic_Rag_MCP/expert-next/next_level.ipynb#L280), khi người dùng nhập "Q3", hệ thống trả về ngay câu trả lời nhưng không lưu cặp tin nhắn này vào [SQLiteSession](file:///G:/RAG/ed/Agentic_Rag_MCP/expert-next/next_level.ipynb#L200). Ở lượt chat tiếp theo, Agent sẽ hoàn toàn mất ngữ cảnh của câu Q3 vừa hỏi.
    Giải pháp: Phải gọi `await convo.add_items([{"role": "user", "content": message}, {"role": "assistant", "content": answer}])` trong khối bypass.
  • Tràn ngữ cảnh khi danh sách câu hỏi FAQ quá dài: Nếu file [faq.jsonl](file:///G:/RAG/ed/Agentic_Rag_MCP/expert-next/knowledge/faq.jsonl) phình to lên 500 câu hỏi, việc nối toàn bộ tiêu đề vào instructions sẽ làm phình Prompt thô đầu vào, tăng chi phí và làm loãng khả năng chú ý của LLM.
    Giải pháp: Dùng bộ tìm kiếm vector nhẹ (hoặc so khớp từ khóa BM25) để lọc ra top 5 câu hỏi FAQ liên quan nhất đến tin nhắn người dùng, và chỉ đưa 5 tiêu đề đó vào instructions của Agent trước mỗi lượt chạy.

Hiểu lầm phổ biến - Common misunderstandings

  • Hiểu lầm rằng MCP server tự động nhúng (embed) dữ liệu: Thực tế, máy chủ `mcp-server-qdrant` mặc định tự sử dụng mô hình nhúng cục bộ `all-MiniLM-L6-v2`. Nếu bạn thay đổi mô hình nhúng ở phía Ingestion mà không cấu hình lại MCP Server, hệ thống sẽ bị lệch vector nhúng (embedding mismatch), dẫn đến việc tìm kiếm ngữ nghĩa không trả về kết quả chính xác.
  • Hiểu lầm rằng SQLiteSession tự động đồng bộ mọi tin nhắn: `SQLiteSession` chỉ tự động lưu các tin nhắn đi qua hàm `Runner.run` hoặc `Runner.run_streamed`. Mọi tin nhắn được sinh ra ngoài luồng này (như xử lý bypass bằng code thuần) bắt buộc phải được lập trình viên bổ sung thủ công vào cơ sở dữ liệu session.
Section 11

Interview Takeaways - Ý chính cho phỏng vấn

Các câu hỏi phỏng vấn hóc búa - Tough questions

  • Làm thế nào để thiết kế một hệ thống RAG vừa đảm bảo trả lời chính xác các câu hỏi FAQ chuẩn của doanh nghiệp, vừa linh hoạt xử lý các câu hỏi mở của khách hàng?
  • Giải thích cách tối ưu hóa chi phí token và độ trễ khi hệ thống phải đối mặt với hàng trăm câu trả lời FAQ rất dài (mỗi câu dài vài trang giấy)?
  • Làm sao để đảm bảo trải nghiệm người dùng không bị gián đoạn (lag/freeze) khi Agent đang thực hiện các cuộc gọi công cụ (tool calls) ngầm có độ trễ lớn?

Khung cấu trúc trả lời phỏng vấn - Answer frame

Khi trả lời phỏng vấn, hãy áp dụng cấu trúc 5 bước chuyên nghiệp:
1. Goal (Mục tiêu): Giảm thiểu độ trễ, tối ưu hóa ngữ cảnh và chi phí token.
2. Mechanism (Cách hoạt động): Tách biệt tiêu đề câu hỏi (đưa vào prompt chỉ mục) và nội dung chi tiết (để ở custom tool). Rẽ nhánh Bypass bằng code lập trình cứng đối với câu hỏi tĩnh.
3. Trade-off (Đánh đổi): Tiết kiệm chi phí token và tăng tốc độ phản hồi đáng kể, nhưng đòi hỏi phải quản lý và đồng bộ bộ nhớ hội thoại thủ công ở nhánh bypass.
4. Pitfalls (Bẫy thực tế): Lỗi đứt gãy mạch hội thoại ngắn hạn nếu quên đồng bộ SQLite Session khi chạy bypass.
5. When to use (Khi nào dùng): Khi xây dựng chatbot chăm sóc khách hàng doanh nghiệp có danh mục FAQ rõ ràng và cần tối ưu hóa chi phí vận hành API.

Interview Q&A / Flashcards - Câu hỏi phỏng vấn / thẻ ôn tập

Thẻ ôn nhanh - Flashcards

  • Q: Kỹ thuật ẩn câu trả lời FAQ chi tiết vào Custom Tool được gọi là gì?
    A: Kỹ thuật tối ưu hóa ngữ cảnh thông qua gọi công cụ động (Context Optimization via Dynamic Tool Lookup).
  • Q: Làm thế nào để tự động focus con trỏ vào ô nhập liệu Gradio sau khi Bot dừng stream?
    A: Sử dụng Javascript `MutationObserver` để theo dõi thuộc tính `disabled` hoặc `readonly` của thẻ `textarea` và gọi hàm `focus()`.
  • Q: Tại sao phải dùng SQLiteSession thay vì lưu lịch sử hội thoại trong biến List của RAM?
    A: Để đảm bảo tính bền vững của dữ liệu hội thoại (persistence). Lịch sử chat sẽ không bị mất khi ứng dụng khởi động lại hoặc server gặp sự cố.

Hỏi đáp tuyển dụng - Interview Q&A

  • Q: Giả sử hệ thống có 10,000 câu FAQ. Làm thế nào để áp dụng thiết kế của expert-next?
    A: Với 10,000 câu, chúng ta không thể nạp hết tiêu đề vào instructions. Cần áp dụng quy trình 2 bước: Bước 1 dùng thuật toán tìm kiếm vector hoặc BM25 siêu nhanh trên Client để tìm ra top 5 câu hỏi liên quan nhất. Bước 2 chỉ nối 5 tiêu đề câu hỏi này kèm số ID tương ứng vào Prompt của Agent để LLM quyết định có nên gọi `faq_tool` hay không. Điều này giữ kích thước Prompt luôn nhỏ và ổn định.
  • Q: Trong file next_level.ipynb, nếu người dùng nhập "Q3" và hệ thống trả kết quả ngay, chuyện gì xảy ra với ngữ cảnh hội thoại ở câu hỏi tiếp theo?
    A: Ở câu hỏi tiếp theo, Agent sẽ bị mất ngữ cảnh vì câu hỏi "Q3" và câu trả lời tương ứng không được lưu vào SQLiteSession. Đây là lỗi logic (bug) nghiêm trọng trong bản code gốc. Để khắc phục, lập trình viên phải ghi nhận dữ liệu thủ công bằng cách sử dụng phương thức `convo.add_items` của SQLiteSession trước khi trả kết quả về cho giao diện chat.
