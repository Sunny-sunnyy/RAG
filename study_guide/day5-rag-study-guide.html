Section 1

Cheat Sheet - Tóm tắt một trang

Phần cần nhớ nhanh trước khi đi vào chi tiết.

Core idea - Ý chính

Advanced RAG là quá trình biến RAG từ một pipeline thử-sai thành một hệ thống tối ưu có đo lường. Bạn tạo baseline, đo retrieval và answer quality, xác định loại câu hỏi fail, áp dụng một kỹ thuật, rồi chạy eval lại để chứng minh cải tiến.

Must know - Cần nhớ

  • Semantic chunking - chia chunk theo ngữ nghĩa: LLM tạo headline, summary, original_text để tăng khả năng retrieval.
  • Query rewriting - viết lại truy vấn: biến câu hỏi thiếu ngữ cảnh thành query ngắn, cụ thể, dễ search.
  • Query expansion - mở rộng truy vấn: search cả câu hỏi gốc và câu hỏi viết lại để giảm rủi ro mất chi tiết.
  • Reranking - tái xếp hạng: dùng LLM/cross-encoder style để sắp lại top chunks theo relevance thật.
  • MRR/nDCG: đo thứ hạng context đúng; Accuracy/Completeness/Relevance: đo chất lượng câu trả lời.

Interview angle - Góc phỏng vấn

Câu trả lời tốt không phải “tôi dùng LangChain/Chroma/OpenAI”, mà là: “Tôi có golden test set, đo retrieval và generation riêng, thấy Basic RAG fail ở spanning/holistic/numerical, nên tôi thêm semantic chunking, query expansion và reranking; sau đó MRR và answer accuracy tăng.”

One-liner - Một câu nhớ: Day 5 dạy bạn xây RAG như một engineer: mỗi kỹ thuật phải gắn với failure mode, metric và bằng chứng cải thiện.
Section 2

Session Overview - Tổng quan buổi học

Day 5 là Pro Day: nâng Basic RAG thành Advanced RAG có kiểm chứng.

Main idea - Ý tưởng chính

Basic RAG lấy top K chunks bằng vector similarity rồi nhét vào prompt. Advanced RAG thêm các bước trước và sau retrieval: chuẩn hóa tài liệu, viết lại query, mở rộng query, sắp xếp lại context và đo lường kết quả.

Why it matters - Vì sao quan trọng

Trong production, failure thường không nằm ở LLM “không thông minh”, mà ở context sai, context thiếu, context bị nhiễu hoặc context đúng nhưng nằm quá thấp trong prompt. Advanced RAG xử lý chính các lỗi này.

Expected outcome - Kết quả mong muốn

Hệ thống Pro giữ cùng interface với app cũ nhưng backend retrieval mạnh hơn: search rộng hơn, context sạch hơn, trả lời chính xác hơn và có dashboard để chứng minh.

Section 5

Core Theory - Lý thuyết cốt lõi

Các technical cluster quan trọng, mỗi cluster gồm concept, methods, tools, trade-off và when to use.

1. Evidence-driven Advanced RAG - RAG nâng cao dựa trên bằng chứng

Concept - Khái niệm

Evidence-driven Advanced RAG là cách phát triển RAG bằng dữ liệu đo lường. Thay vì hỏi “kỹ thuật nào đang hot?”, bạn hỏi “metric nào đang thấp, failure mode nằm ở đâu, và kỹ thuật nào có khả năng cải thiện đúng điểm đó?”.

Methods / Techniques - Phương pháp / kỹ thuật

  • Baseline measurement - đo baseline trước khi tối ưu.
  • Single-change experiment - chỉ đổi một biến mỗi lần.
  • Category analysis - phân tích theo loại câu hỏi.
  • Regression tracking - kiểm tra kỹ thuật mới không làm hỏng nhóm câu hỏi cũ.

Tools / Models / Frameworks - Công cụ / mô hình / framework

evaluation/tests.jsonl, evaluation/eval.py, evaluator.py, evaluator2.py, LiteLLM, Pydantic structured outputs.

Trade-off - Đánh đổi

Có eval loop thì chậm hơn ở giai đoạn development, nhưng giảm rủi ro tối ưu mù. Không có eval loop, bạn dễ thêm reranking/query expansion rồi tưởng là tốt chỉ vì demo vài câu đẹp.

When to use - Khi nào nên dùng: luôn dùng khi RAG phục vụ dữ liệu nội bộ, thông tin chính xác hoặc sản phẩm có người dùng thật.

2. Semantic chunking - Chia chunk theo ngữ nghĩa

Concept - Khái niệm

Semantic chunking dùng LLM đọc tài liệu và chia thành các mảnh có nghĩa, thay vì cắt theo số ký tự cố định. Trong code Day 5, mỗi chunk gồm headline, summary và original_text.

Methods / Techniques - Phương pháp / kỹ thuật

  • LLM-based chunking - dùng LLM chia tài liệu thành chunk.
  • Overlap - gối đầu khoảng 25% hoặc khoảng 50 từ để tránh mất ngữ cảnh.
  • Document pre-processing - viết lại bảng/số liệu thành mô tả dễ search.
  • Chunk enrichment - thêm headline và summary vào nội dung nhúng vector.

Tools / Models / Frameworks - Công cụ / mô hình / framework

pro_implementation/ingest.py, LiteLLM completion, Pydantic Chunk/Chunks, OpenAI model cho chunking.

Trade-off - Đánh đổi

Chất lượng retrieval tốt hơn, nhưng ingestion tốn token, mất thời gian và có thể gặp rate limit. Nếu dữ liệu rất lớn hoặc thay đổi liên tục, semantic chunking cần queue, cache và retry nghiêm túc.

When to use - Khi nào nên dùng: khi tài liệu có cấu trúc không đều, nhiều bảng, nhiều đoạn ngắn, hoặc Basic RAG hay fail vì chunk không chứa đủ ngữ cảnh.

3. Native vector store - Kho vector native không qua LangChain wrapper

Concept - Khái niệm

Day 5 thay abstraction LangChain bằng các thư viện trực tiếp: tự gọi OpenAI embeddings, tự tạo Chroma collection, tự add ids/embeddings/documents/metadatas. Điều này làm pipeline rõ hơn và dễ debug hơn.

Methods / Techniques - Phương pháp / kỹ thuật

  • Persistent collection - lưu Chroma trong preprocessed_db.
  • Delete and rebuild - xóa collection cũ khi rebuild embeddings.
  • Metadata preservation - giữ source và type để show context/citation.
  • Embedding consistency - dùng cùng embedding model khi ingest và query.

Tools / Models / Frameworks - Công cụ / mô hình / framework

ChromaDB PersistentClient, OpenAI text-embedding-3-large, NumPy, Plotly, t-SNE.

Trade-off - Đánh đổi

Native code ít magic hơn và dễ kiểm soát hơn, nhưng bạn tự chịu trách nhiệm về retry, batching, observability, schema, cleanup và backward compatibility.

When to use - Khi nào nên dùng: khi cần kiểm soát production behavior, debug chi tiết hoặc tối ưu từng bước pipeline.

4. Query rewriting & query expansion - Viết lại và mở rộng truy vấn

Concept - Khái niệm

Query rewriting biến câu hỏi người dùng thành một query ngắn, cụ thể, dễ tìm trong knowledge base. Query expansion chạy search trên nhiều phiên bản của câu hỏi để không phụ thuộc vào một cách diễn đạt duy nhất.

Methods / Techniques - Phương pháp / kỹ thuật

  • History-aware rewriting - dùng chat history để giải nghĩa câu hỏi mơ hồ.
  • Original + rewritten search - search cả câu gốc và câu rewritten.
  • Merge and deduplicate - gộp kết quả, bỏ trùng theo page_content.
  • Short refined query - ép LLM chỉ trả về query, không trả lời dài dòng.

Tools / Models / Frameworks - Công cụ / mô hình / framework

rewrite_query, fetch_context_unranked, merge_chunks, LiteLLM, OpenAI embeddings, Chroma query.

Trade-off - Đánh đổi

Rewriting có thể giúp câu hỏi mơ hồ trở nên searchable, nhưng cũng có thể gây query dilution - loãng truy vấn. Module final giảm rủi ro này bằng cách search cả original question và rewritten question.

When to use - Khi nào nên dùng: khi người dùng hỏi follow-up, câu hỏi thiếu entity, hoặc Basic RAG fail vì query không khớp cách tài liệu diễn đạt.

5. Reranking - Tái xếp hạng kết quả truy xuất

Concept - Khái niệm

Vector search nhanh nhưng thô. Reranking đọc lại các candidate chunks cùng với câu hỏi và sắp xếp lại theo relevance thật. Nó giống cross-encoder style: xét cặp question + chunk thay vì so khoảng cách vector độc lập.

Methods / Techniques - Phương pháp / kỹ thuật

  • Retrieve broad - lấy nhiều candidate hơn, ví dụ RETRIEVAL_K = 20.
  • Structured rank output - ép LLM trả về danh sách ID theo schema RankOrder.
  • Final top K - chỉ đưa top chunks sau rerank vào prompt, ví dụ FINAL_K = 10.
  • Lost-in-the-middle mitigation - đẩy context đúng lên đầu để LLM dễ dùng.

Tools / Models / Frameworks - Công cụ / mô hình / framework

rerank, Pydantic RankOrder, LiteLLM response format, LLM ranker hoặc cross-encoder chuyên dụng.

Trade-off - Đánh đổi

Reranking thường tăng MRR và giảm nhiễu context, nhưng thêm 1 lần gọi model, tăng latency và token cost. Nếu top K ban đầu không chứa context đúng, reranking không thể tự tạo ra thông tin đúng.

When to use - Khi nào nên dùng: khi retrieval recall khá ổn nhưng context đúng thường nằm vị trí 4-20 thay vì top 1-3.

6. Evaluation metrics - Chỉ số đánh giá

Concept - Khái niệm

Retrieval metrics đo hệ thống có tìm đúng context không. Answer metrics đo LLM có trả lời chính xác, đầy đủ và đúng trọng tâm không. Hai nhóm metric này phải tách riêng để biết lỗi nằm ở retrieval hay generation.

Methods / Techniques - Phương pháp / kỹ thuật

  • MRR - Mean Reciprocal Rank: điểm cao khi keyword/context đúng nằm gần top 1.
  • nDCG - Normalized Discounted Cumulative Gain: đo chất lượng thứ tự của nhiều kết quả liên quan.
  • Keyword coverage - độ phủ từ khóa: phần trăm keywords xuất hiện trong top results.
  • LLM-as-a-judge: chấm accuracy, completeness, relevance theo thang 1-5.

Tools / Models / Frameworks - Công cụ / mô hình / framework

RetrievalEval, AnswerEval, calculate_mrr, calculate_ndcg, Pydantic, LiteLLM.

Trade-off - Đánh đổi

MRR/nDCG dựa trên keyword có thể bỏ sót relevance ngữ nghĩa. LLM judge linh hoạt hơn nhưng có bias, chi phí và biến động. Vì vậy nên dùng cả hai.

When to use - Khi nào nên dùng: dùng retrieval metrics khi debug context; dùng answer metrics khi đánh giá trải nghiệm cuối.

Section 6

Workflow / Pipeline - Quy trình / luồng hoạt động

Toàn bộ quá trình xây hệ thống Advanced RAG từ dữ liệu thô đến câu trả lời và dashboard.

System architecture - Kiến trúc hệ thống

1
Knowledge base - Kho tri thức

Documents nằm trong knowledge-base, chia theo company, products, contracts, employees. Dữ liệu là Markdown nên giữ được heading/list/table tốt hơn plain text.

2
Pro ingestion - Nạp dữ liệu nâng cao

fetch_documents quét file Markdown, process_document gọi LLM tạo semantic chunks có headline/summary/original_text, create_embeddings lưu vector vào Chroma preprocessed_db.

3
Query transformation - Biến đổi truy vấn

rewrite_query dùng LLM tạo refined query ngắn và cụ thể hơn. Module final vẫn giữ original question để tránh mất chi tiết quan trọng.

4
Broad retrieval - Truy xuất rộng

fetch_context_unranked embed câu hỏi bằng text-embedding-3-large, query Chroma lấy top 20 chunks cho câu gốc và câu rewritten.

5
Merge, deduplicate, rerank - Gộp, khử trùng, tái xếp hạng

merge_chunks bỏ duplicate theo content; rerank dùng LLM ranker trả về thứ tự chunk IDs; hệ thống lấy top 10 chunk sau rerank.

6
Grounded generation - Sinh câu trả lời có căn cứ

make_rag_messages bơm context kèm source vào system prompt; answer_question gọi model trả lời, trả về cả answer và context cho UI.

7
Evaluation loop - Vòng lặp đánh giá

evaluation/eval.py chạy golden tests, đo MRR/nDCG/coverage và dùng LLM judge chấm Accuracy/Completeness/Relevance. Dashboard trong evaluator.py và evaluator2.py hiển thị kết quả.

Key design choice - Quyết định thiết kế quan trọng: UI Gradio không cần biết pipeline bên dưới thay đổi. Nó vẫn gọi answer_question(last_message, prior). Đây là abstraction tốt: contract giữ nguyên, implementation có thể nâng cấp từ Basic RAG sang Pro RAG.
Section 7

Techniques - Kỹ thuật sử dụng

Danh sách technique Day 5, kèm vấn đề giải quyết và điều kiện dùng.

Technique - Kỹ thuật Problem - Vấn đề How it works - Cách hoạt động Trade-off - Đánh đổi When to use - Khi nào nên dùng
Chunking R&D - Thử nghiệm chunking Chunk quá ngắn thiếu context, chunk quá dài gây nhiễu. Thử nhiều chunk size/overlap/semantic boundary và đo lại MRR/nDCG. Tốn thời gian rebuild vector store. Khi retrieval thất thường hoặc câu trả lời thiếu chi tiết.
Encoder R&D - Thử nghiệm embedding model Embedding model không biểu diễn tốt domain hoặc ngôn ngữ. Đổi encoder, rebuild DB, đo lại trên golden test set. Dimension mismatch, cost và latency khác nhau. Khi keyword đúng không xuất hiện dù chunking ổn.
Improve prompts - Cải thiện prompt LLM trả lời lan man, không đủ, hoặc không bám context. Thêm tiêu chí accuracy/relevance/completeness, source context, history và rule “If you don't know, say so”. Prompt dài hơn, dễ overfit nếu quá cứng. Khi retrieval đúng nhưng answer score thấp.
Document pre-processing - Tiền xử lý tài liệu Bảng, số liệu, PDF/Word hoặc cấu trúc khó embed. Chuyển sang Markdown, viết lại phần khó thành văn bản mô tả dễ search. Tốn token, có rủi ro LLM làm sai nếu không giữ original_text. Khi tài liệu nhiều bảng, điều khoản, số liệu hoặc format nhiễu.
Query rewriting - Viết lại truy vấn User hỏi follow-up hoặc dùng đại từ như “nó”, “người đó”. LLM tạo refined query ngắn dựa trên question và history. Có thể làm loãng hoặc bỏ mất chi tiết. Khi chatbot có multi-turn conversation.
Query expansion - Mở rộng truy vấn Một cách diễn đạt không tìm đủ context. Search nhiều query, merge/deduplicate rồi rerank. Tăng số query, cost và latency. Khi câu hỏi spanning, relationship hoặc có nhiều cách gọi entity.
Re-ranking - Tái xếp hạng Context đúng có trong top 20 nhưng không nằm top 1-3. LLM/cross-encoder style đọc chunks và trả về thứ tự relevance. Thêm model call, tăng latency. Khi MRR thấp nhưng keyword coverage tương đối ổn.
Hierarchical RAG - RAG phân cấp Câu hỏi holistic cần tổng hợp toàn corpus. Tạo summaries nhiều cấp: document summary, section summary, chunk detail. Complexity cao, cần bảo trì summary index. Khi hỏi tổng số, xu hướng, portfolio, thống kê toàn bộ.
GraphRAG - RAG dựa trên đồ thị Quan hệ entity phức tạp, nhiều liên kết gián tiếp. Trích entity/relation, xây graph, retrieve theo neighborhood. Pipeline nặng, cần entity resolution tốt. Khi dữ liệu có network rõ: người, công ty, hợp đồng, sản phẩm.
Agentic RAG - RAG tác nhân Một retriever không đủ: cần SQL, grep, vector, web, memory. Agent chọn tool, tự đánh giá, có thể search lại hoặc sửa câu trả lời. Khó kiểm soát, cần guardrails và observability. Khi query routing phức tạp và cần multi-tool retrieval.

Implementation Insights - Góc nhìn triển khai

Phân tích các file code chính và cách chúng ghép thành hệ thống.

Ingestion module - Module nạp dữ liệu

  • fetch_documents: tự thay thế LangChain DirectoryLoader, quét Markdown trong knowledge-base.
  • Chunk: Pydantic model ép LLM trả về headline, summary, original_text.
  • create_chunks: dùng multiprocessing Pool để xử lý documents song song.
  • create_embeddings: gọi OpenAI embeddings, tạo Chroma collection mới trong preprocessed_db.

Answer module - Module trả lời

  • rewrite_query: tạo refined query ngắn để search knowledge base.
  • fetch_context_unranked: vectorize query và lấy top 20 chunks từ Chroma.
  • merge_chunks: gộp kết quả từ original và rewritten query, bỏ trùng.
  • rerank: LLM ranker trả về thứ tự chunk IDs qua schema RankOrder.
  • answer_question: lấy final context, tạo RAG messages, gọi model sinh answer.

UI integration - Tích hợp giao diện

app.py chỉ đổi import từ basic sang pro: from pro_implementation.answer import answer_question. Hàm chat vẫn lấy last message, prior history, gọi answer_question và render retrieved context.

Evaluation integration - Tích hợp đánh giá

evaluation/eval.py import answer_question và fetch_context từ pro implementation. Điều này cho phép cùng bộ tests đo bản basic hoặc bản pro chỉ bằng đổi import.

Key code path - Đường đi chính trong code

question
  -> rewrite_query(question)
  -> fetch_context_unranked(original_question)
  -> fetch_context_unranked(rewritten_question)
  -> merge_chunks(chunks_from_original, chunks_from_rewritten)
  -> rerank(original_question, merged_chunks)
  -> make_rag_messages(question, history, top_reranked_chunks)
  -> completion(model=MODEL, messages=messages)
  -> answer, context
Implementation nuance - Chi tiết cần để ý: Notebook Day 5 ban đầu dùng rewritten query làm query chính. Module final trong pro_implementation/answer.py cải tiến hơn: search cả câu hỏi gốc và câu rewritten. Đây là thiết kế tốt hơn vì chống mất thông tin do query rewriting.
Section 10

Evidence-driven Loop - Vòng lặp Advanced RAG dựa trên bằng chứng

Day 5 không chỉ build pipeline, mà còn chứng minh pipeline tốt hơn bằng metric.

Golden dataset - Bộ câu hỏi vàng

evaluation/tests.jsonl chứa câu hỏi, keywords cần xuất hiện, reference answer và category. Bộ này không chỉ có direct facts mà còn có temporal, comparative, numerical, relationship, spanning và holistic.

Ý nghĩa: nếu chỉ test direct facts, Basic RAG có thể trông ổn. Spanning/holistic/numerical mới là nơi pipeline thật sự lộ điểm yếu.

Metric journey - Hành trình chỉ số

  • Basic RAG - Bản cơ bản: MRR khoảng 0.73, Accuracy khoảng 3.99/5.
  • Embedding/chunk tuning - Tối ưu encoder/chunk: MRR khoảng 0.79, Accuracy khoảng 4.21/5.
  • Advanced RAG Pro - Bản nâng cao: MRR khoảng 0.91, Accuracy khoảng 4.62/5.

Optimization workflow - Workflow tối ưu

  1. Curate test set - Tạo bộ test: câu hỏi thật, expected keywords, reference answers, categories.
  2. Run baseline - Chạy baseline: đo MRR/nDCG/coverage và answer scores.
  3. Find weakness - Tìm điểm yếu: xem category nào đỏ/vàng, câu nào fail, context đúng có xuất hiện không.
  4. Apply one technique - Áp dụng một kỹ thuật: ví dụ semantic chunking, query expansion hoặc reranking.
  5. Re-run eval - Chạy lại đánh giá: so sánh số mới với baseline, kiểm tra regression.
  6. Promote or rollback - Giữ hoặc bỏ: chỉ giữ kỹ thuật nếu metric và case review chứng minh cải thiện.
How to read metric conflicts - Cách đọc khi metric mâu thuẫn: Nếu MRR cao nhưng Accuracy thấp, retrieval có thể đúng nhưng generator hoặc prompt đang yếu. Nếu MRR thấp nhưng Accuracy cao, LLM có thể đang đoán bằng parametric memory; đây là rủi ro vì dữ liệu nội bộ không nằm trong training data.
Section 11

Pitfalls - Lỗi / bẫy thường gặp

Các lỗi dễ gặp khi xây Advanced RAG và cách nhận diện.

Failure modes - Trường hợp dễ fail

  • Query dilution - Loãng truy vấn: rewritten query bỏ mất entity hoặc con số quan trọng. Giảm bằng cách search cả original query.
  • Context contamination - Nhiễu context: top K chứa nhiều người/sản phẩm/hợp đồng tương tự, LLM trộn thông tin.
  • Lost in the middle - Mất thông tin giữa prompt: context đúng có nhưng nằm quá thấp, LLM bỏ qua. Reranking giúp đẩy lên đầu.
  • Embedding dimension mismatch - Lệch chiều vector: đổi embedding model mà không rebuild DB sẽ crash hoặc query sai.
  • Rate limit during ingestion - Giới hạn API khi ingest: semantic chunking song song cần retry/backoff và giảm workers khi cần.
  • Overfitting to eval set - Quá khớp bộ test: tối ưu để thắng tests.jsonl nhưng fail ở câu hỏi người dùng thật.

Common misunderstandings - Hiểu lầm phổ biến

  • “Vector DB xịn là RAG xịn”: sai. Chất lượng retrieval phụ thuộc nhiều vào data prep, chunking, encoder, query và rerank.
  • “Context window lớn sẽ giết RAG”: sai. Context window lớn không giải quyết cost, latency, privacy, source control và attention noise.
  • “Reranking sửa được mọi thứ”: sai. Nếu retrieval không lấy được context đúng trong candidate set, reranker không có gì để đẩy lên.
  • “LLM judge là chân lý”: sai. LLM judge có bias, nên cần prompt rõ, temperature thấp, structured output và review sample thủ công.
  • “No LangChain nghĩa là LangChain tệ”: sai. Bài học là hiểu abstraction boundary. LangChain tốt cho speed; native tốt cho control.
  • “Thêm technique luôn tốt hơn”: sai. Technique không gắn metric chỉ làm hệ thống chậm, đắt và khó debug hơn.
Code review note - Ghi chú review code: evaluator2.py có đoạn cố cấu hình answer_module.llm theo style cũ, trong khi pro_implementation/answer.py hiện dùng LiteLLM completion. Nếu chạy evaluator2, cần kiểm tra lại assumption này vì có thể là legacy code không còn tác dụng.

Answer frame - Khung trả lời

Section 13

Interview Q&A / Flashcards - Câu hỏi phỏng vấn / thẻ ôn tập

Ôn nhanh và luyện trả lời phỏng vấn.

Flashcards - Thẻ ôn nhanh

  • Q: Advanced RAG là gì? A: RAG có thêm data prep, query transformation, reranking và evaluation loop để cải thiện bằng metric.
  • Q: Semantic chunking khác recursive chunking thế nào? A: Semantic chunking cắt theo ý nghĩa bằng LLM; recursive chunking cắt theo separator/ký tự.
  • Q: Vì sao mỗi chunk có headline và summary? A: Để embedding chứa tín hiệu ngữ nghĩa rõ hơn, giúp retrieval match câu hỏi tốt hơn.
  • Q: MRR đo gì? A: Vị trí của kết quả đúng đầu tiên; rank 1 được 1 điểm, rank 2 được 0.5 điểm.
  • Q: nDCG đo gì? A: Chất lượng thứ tự của các kết quả liên quan trong top K, có discount theo vị trí.
  • Q: Query expansion giải quyết gì? A: Giảm rủi ro một cách diễn đạt duy nhất không retrieve đủ context.
  • Q: Reranking cần điều kiện gì để hiệu quả? A: Candidate set ban đầu phải có chứa context đúng.
  • Q: Vì sao dùng structured outputs? A: Để LLM trả về JSON/schema ổn định, máy có thể parse và tự động hóa pipeline.

Interview Q&A - Hỏi đáp phỏng vấn

Q1: Hãy thiết kế một Advanced RAG pipeline cho dữ liệu nội bộ công ty.

Tôi sẽ bắt đầu bằng knowledge base chuẩn hóa sang Markdown, sau đó chạy ingestion pipeline tạo semantic chunks bằng LLM, mỗi chunk có headline, summary và original text. Tôi embed chunks bằng một embedding model nhất quán và lưu vào Chroma. Khi user hỏi, tôi rewrite query dựa trên history, search cả original và rewritten query, merge/deduplicate, rerank top candidates, rồi đưa top context vào prompt. Cuối cùng tôi đo retrieval bằng MRR/nDCG/coverage và đo answer bằng LLM-as-a-judge trên accuracy, completeness, relevance.

Q2: Khi nào bạn thêm reranking?

Tôi thêm reranking khi recall/coverage cho thấy context đúng có xuất hiện trong top 10-20 nhưng MRR thấp, tức là context đúng nằm quá sâu. Reranking không phải bước đầu tiên nếu retrieval không tìm được tài liệu đúng. Nó phù hợp khi cần sub-select context tốt nhất trước khi đưa vào LLM để giảm nhiễu và lost-in-the-middle.

Q3: Query rewriting có nguy hiểm gì?

Query rewriting có thể làm mất entity, số liệu, thời gian hoặc điều kiện trong câu hỏi gốc. Đó là query dilution. Cách giảm rủi ro là search cả original query và rewritten query, sau đó merge/deduplicate rồi rerank theo original question.

Q4: Vì sao không dùng LangChain trong Pro Day?

Không dùng LangChain không có nghĩa LangChain tệ. Mục tiêu là hiểu động cơ bên dưới: loader, chunker, embedding call, vector store query, prompt construction, reranking và evaluation. Native pipeline giúp kiểm soát latency, retry, schema và debug dễ hơn. Khi cần connector nhanh hoặc agent ecosystem, LangChain vẫn là lựa chọn hợp lý.

Q5: Nếu accuracy thấp nhưng retrieval metrics cao, bạn xử lý thế nào?

Tôi sẽ kiểm tra prompt, độ dài context, thứ tự context và generation model. Retrieval cao nghĩa là context đúng đã vào pipeline, nhưng generator có thể bị nhiễu, bỏ qua context, hoặc trả lời lan man. Cách xử lý: giảm final K, cải thiện prompt, thêm source formatting, hoặc dùng model generation mạnh hơn.
