Section 5

Core Theory - Lý thuyết cốt lõi

MCP - Model Context Protocol

Concept - Khái niệm: chuẩn giao tiếp để tool/server có thể expose capabilities cho model hoặc agent theo cách nhất quán.

Methods / Techniques - Phương pháp / kỹ thuật: chạy MCP server qua STDIO, liệt kê tools, cấp các server đó cho agent.

Tools / Frameworks - Công cụ / framework: MCPServerStdio, mcp-server-fetch, mcp-server-qdrant.

Trade-off - Đánh đổi: tái sử dụng nhanh nhưng phụ thuộc chất lượng tool server và schema đã có.

When to use - Khi nào nên dùng: khi bạn muốn tích hợp tools nhanh và chuẩn hóa thay vì tự viết glue code riêng cho từng tool.

Agentic RAG - RAG có tác tử

Concept - Khái niệm: agent được cấp retrieval capabilities và tự quyết định khi nào cần dùng chúng.

Methods / Techniques - Phương pháp / kỹ thuật: natural-language tasking, multi-turn tool use, memory lookup trước khi answer.

Tools / Models / Frameworks: OpenAI Agents SDK, LiteLLM-backed model, Qdrant MCP tool.

Trade-off - Đánh đổi: linh hoạt hơn code cứng, nhưng khó kiểm soát hơn và dễ gọi tool không tối ưu.

When to use - Khi nào nên dùng: khi retrieval không còn là bước luôn cố định và cần cho agent quyền điều phối.

Long-term memory - Bộ nhớ dài hạn

Concept - Khái niệm: facts hoặc memories được nhúng vector và lưu trong vector store để truy hồi về sau.

Methods / Techniques - Phương pháp / kỹ thuật: ingest từ web, deduplicate, store payload vào collection knowledge.

Tools / Services - Công cụ / dịch vụ: Qdrant local path, sentence-transformer mặc định của MCP server.

Trade-off - Đánh đổi: useful cho recall dài hạn nhưng chất lượng phụ thuộc vào dữ liệu ingest và query formulation.

When to use - Khi nào nên dùng: khi tri thức domain cần tồn tại qua nhiều phiên chat.

Short-term memory - Bộ nhớ ngắn hạn

Concept - Khái niệm: lưu conversation history để agent hiểu follow-up questions và context đang diễn ra.

Methods / Techniques - Phương pháp / kỹ thuật: dùng SQLiteSession truyền vào Runner.run hoặc run_streamed.

Tools / Frameworks: OpenAI Agents SDK sessions, SQLite-backed persistence.

Trade-off - Đánh đổi: tốt cho hội thoại nhưng không thay thế knowledge base dài hạn.

When to use - Khi nào nên dùng: gần như luôn dùng cho chatbot có nhiều lượt trao đổi.

Concept relationship - Quan hệ giữa khái niệm: MCP cung cấp tools. Agent dùng tools để ingest và retrieve. Qdrant giữ long-term memory. SQLiteSession giữ short-term memory. Khi thêm faq_tool, hệ thống chuyển từ vector-only retrieval sang hybrid retrieval - truy hồi lai.
Section 6

Workflow / Pipeline - Quy trình / luồng hoạt động

High-level flow - Luồng tổng quát

  1. Chọn model và test agent tối thiểu.
  2. Khai báo URLs làm nguồn knowledge.
  3. Mở fetch MCP server và qdrant MCP server.
  4. Tạo ingest agent với instruction lưu memories duy nhất.
  5. Chạy loop qua nhiều URL để populate vector store.
  6. Kiểm tra collection trong Qdrant.
  7. Tạo expert chat agent chỉ dùng memory tools để trả lời.
  8. Ở bản nâng cấp, thêm FAQ tool và streamed chat UX.

Data / control flow - Luồng dữ liệu / điều khiển

Input của pha ingest là website URLs. Fetch tool lấy nội dung web. Agent đọc nội dung, tìm facts quan trọng, kiểm tra duplicate bằng memory lookup, rồi dùng Qdrant tool để store. Output là collection knowledge.

Ở pha chat, input là user message. Agent dùng vector memory để retrieve context. Ở next_level.ipynb, nếu câu hỏi map vào FAQ thì có thể gọi faq_tool; nếu người dùng nhập dạng Q2, hệ thống bypass model và trả lời trực tiếp, nhưng vẫn ghi vào session để giữ continuity.

Pipeline summary - Tóm tắt pipeline: website -> fetch -> ingest agent -> qdrant-store / qdrant-find -> local vector DB -> expert agent -> SQLiteSession -> Gradio chat UI.
Section 7

Techniques - Kỹ thuật sử dụng

Tool-based ingestion - ingest bằng tool

Agent không được đưa raw documents sẵn trong prompt; thay vào đó nó chủ động fetch và store bằng tools. Đây là một pattern quan trọng vì retrieval/storage được tách khỏi prompt text thuần.

Dedup-aware memory population - nạp memory có kiểm tra trùng

Instruction yêu cầu agent check memory trước khi thêm mới. Điều này dạy một mindset đúng: ingest không chỉ là append dữ liệu, mà còn phải quan tâm noise và duplicate.

Hybrid retrieval - truy hồi lai

Bản next_level dùng cả vector memory và deterministic FAQ tool. Đây là một cải tiến rất thực dụng vì common questions không nên phụ thuộc hoàn toàn vào semantic search.

Streaming - phản hồi theo luồng

Dùng Runner.run_streamed và event parsing để hiển thị token output cùng trạng thái gọi tool. Đây là cải tiến UX chứ không phải retrieval improvement, nhưng rất đáng giá.

Section 9

Implementation Insights - Góc nhìn triển khai

What the code does - Code đang làm gì

ingest.ipynb làm ba việc chính: chọn model, tạo local vector DB path, rồi chạy ingest agent để đổ knowledge từ web vào Qdrant. Sau đó notebook chuyển sang pha chat bằng một hàm chat() chỉ vài dòng nhưng đã có model + memory + session.

next_level.ipynb mở rộng bằng cách đọc knowledge/faq.jsonl, dựng faq_tool, thêm streaming event handling, và thêm đường tắt direct answer cho các câu kiểu Q2.

Why this design - Vì sao chọn thiết kế này

Thiết kế này tối ưu cho teaching speed và concept clarity. Người học thấy ngay: retrieval có thể là tool, memory có thể là MCP server, session là short-term memory, và UI có thể ghép rất nhanh.

Trade-off là hệ thống này intentionally đơn giản: chưa có eval harness riêng, chưa có ingestion quality control sâu, chưa có router phức tạp, và domain data source còn nhỏ. Nhưng đó lại là điểm mạnh cho việc học nền tảng.

Code map - Bản đồ file code quan trọng
  • ingest.ipynb: bài gốc, dựng ingest flow và chat flow cơ bản.
  • next_level.ipynb: thêm FAQ tool, streaming, direct answer path.
  • knowledge/faq.jsonl: FAQ data có cấu trúc, 58 dòng JSONL.
  • FAQ.md: spec để tạo hoặc cập nhật file FAQ JSONL.
  • styles.py: CSS/JS cho Gradio, tập trung UX chứ không đổi logic RAG.
  • pyproject.toml: xác nhận stack Python, Gradio, OpenAI Agents SDK, Qdrant client.
Notable implementation details - Chi tiết dễ bị bỏ qua
  • list_tools() trước khi dùng: đây là bước tốt để inspect capabilities thật sự của MCP server.
  • max_turns=50 trong ingest: cho agent đủ chỗ để fetch, check memory, rồi store nhiều lần.
  • Instant answer path phải ghi vào session: nếu không thêm await convo.add_items(...) thì lịch sử hội thoại sẽ bị lệch.
  • knowledge/vectordb bị ignore: vector DB là generated artifact, không phải source file commit cùng repo.
Section 10

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - Trường hợp dễ fail

  • Weak ingestion - ingest yếu: web content nghèo hoặc instruction không rõ sẽ làm memories nông và không hữu ích.
  • Duplicate/noisy memory: nếu agent hoặc tool không kiểm tra kỹ, vector DB dễ đầy facts trùng hoặc quá nhỏ lẻ.
  • Over-trusting vector search: câu hỏi FAQ có thể retrieve không ổn định nếu chỉ dựa vào semantic memory.
  • Session inconsistency: bypass model nhưng không ghi session sẽ làm follow-up bị mất ngữ cảnh.
  • Environment friction: README cảnh báo Windows có thể phải dùng WSL2 để MCP chạy ổn định.

Common misunderstandings - Hiểu lầm phổ biến

  • "MCP là vector DB": sai; MCP là protocol/layer expose tools, còn Qdrant chỉ là một server/tool cụ thể.
  • "Long-term memory thay thế short-term memory": sai; một cái giữ domain knowledge, một cái giữ hội thoại hiện tại.
  • "Có nhiều tools thì agent chắc chắn tốt hơn": sai; nhiều tools hơn cũng tăng ambiguity, latency và cost.
  • "Demo chạy nhanh tức là production-ready": sai; đây là demo học kiến trúc, chưa phải hardened system.
