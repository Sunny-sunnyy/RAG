Session Overview - Tổng quan buổi học

Buổi này thực sự dạy gì?

Buổi này dạy cách suy nghĩ về RAG như một decision system - hệ thống ra quyết định, không chỉ là một pipeline tuyến tính. Nếu Day 5 dạy bạn tối ưu retrieval bằng nhiều kỹ thuật, thì repo này dạy bạn câu hỏi quan trọng hơn: khi nào nên dùng kỹ thuật nào?

Section 5

Core Theory - Lý thuyết cốt lõi

1. Hierarchical parent-child chunking - Chia chunk phân cấp cha-con

Concept - Khái niệm: dùng child chunks làm index layer - lớp chỉ mục để match chi tiết nhỏ; dùng parent chunks làm context layer - lớp ngữ cảnh để cung cấp đầy đủ thông tin khi LLM trả lời.

Methods / Techniques - Phương pháp / kỹ thuật: LLM semantic chunking, explicit entity rewriting trong chunk, lưu parent vào SQLite, lưu child embeddings vào Chroma.

Tools / Models / Frameworks / Services - Công cụ / mô hình / framework / dịch vụ: OpenAI embeddings, LiteLLM structured output, ChromaDB, SQLite, Pydantic.

Trade-off - Đánh đổi: ingestion đắt hơn, schema phức tạp hơn, nhưng retrieval chính xác hơn và generation grounded hơn.

When to use - Khi nào nên dùng: khi dữ liệu có nhiều câu hỏi fact nhỏ nhưng câu trả lời cuối vẫn cần ngữ cảnh rộng hoặc đầy đủ.

2. Query router agent - Tác tử định tuyến truy vấn

Concept - Khái niệm: hệ thống không assume mọi câu hỏi giống nhau; nó dùng một LLM router để chọn tool retrieval tốt nhất cho từng loại query.

Methods / Techniques - Phương pháp / kỹ thuật: function calling, intent classification bằng prompt, tool schema chặt chẽ, single strategy selection.

Tools / Models / Frameworks / Services - Công cụ / mô hình / framework / dịch vụ: LiteLLM tool calling, Pydantic, custom retrieval functions.

Trade-off - Đánh đổi: tăng latency và prompt complexity; nếu router chọn sai tool thì retrieval cả pipeline có thể lệch.

When to use - Khi nào nên dùng: khi test set hoặc user traffic có nhiều query classes với failure modes khác nhau.

3. Query decomposition - Phân rã truy vấn cho multi-hop

Concept - Khái niệm: một câu hỏi spanning thường ẩn nhiều sub-problems; decomposition biến nó thành các sub-queries ngắn và searchable.

Methods / Techniques - Phương pháp / kỹ thuật: LLM tạo danh sách sub-queries độc lập, search song song theo từng query, merge kết quả, rerank theo câu hỏi gốc.

Tools / Models / Frameworks / Services - Công cụ / mô hình / framework / dịch vụ: LiteLLM structured output, vector search, reranker.

Trade-off - Đánh đổi: recall tăng nhưng candidate set lớn hơn, cost cao hơn, dễ noisy hơn nếu decomposition kém.

When to use - Khi nào nên dùng: khi answer cần nối nhiều facts rải rác ở nhiều tài liệu.

4. Self-reflective evaluator agent - Tác tử tự đánh giá câu trả lời

Concept - Khái niệm: sau khi generate answer lần đầu, hệ thống tự kiểm tra xem answer có thiếu, vague hay nói “không tìm thấy” không. Nếu có thì tự tạo refined query để retry.

Methods / Techniques - Phương pháp / kỹ thuật: LLM-as-a-judge trong runtime, controlled retry, critique-driven query refinement.

Tools / Models / Frameworks / Services - Công cụ / mô hình / framework / dịch vụ: LiteLLM structured output, Pydantic schema RetryEvaluation.

Trade-off - Đánh đổi: tăng robustness nhưng cũng tăng latency và chi phí; phải giới hạn số lần retry để tránh loop.

When to use - Khi nào nên dùng: khi bài toán đòi độ đầy đủ cao, và có nhiều câu hỏi mà pass đầu tiên thường thiếu một nửa thông tin.

5. ingest.py implementation map - Implementation-complete view
  • Environment and config: load_dotenv(override=True), OpenAI client, exponential retry, MODEL = "openai/gpt-4.1-nano", AVERAGE_PARENT_CHUNK_SIZE = 1000, WORKERS = 3, embedding_model = "text-embedding-3-large".
  • Dual-store layout: parent context is stored in parent_chunks.db; child vectors are stored in Chroma preprocessed_hierarchic_db.
  • SQLite schema creation: init_parent_chunk_db() creates table parent_chunks(parent_id, parent_headline, parent_chunk, metadata).
  • Document loading: fetch_documents() scans every knowledge-base subfolder, reads every .md file, and normalizes each record to {type, source, text}.
  • Structured contracts: Result, Chunk, and Chunks define the full ingestion contract.
  • Prompt constraints: make_prompt() enforces parent/child hierarchy, 10-20% overlap, full-document coverage, child length 1-2 sentences, and explicit entity replacement for pronouns.
  • Chunk processing: process_document() uses LiteLLM with response_format=Chunks, validates via Pydantic, then converts each item to Result.
  • Parallel ingestion: create_chunks() uses multiprocessing.Pool plus tqdm; the code explicitly notes reducing workers if rate limits appear.
  • Parent linkage: assign_parent_ids() writes metadata["parent_id"] after chunking finishes.
  • Parent persistence: insert_parent_chunk() stores all parent chunks with INSERT OR REPLACE and JSON metadata.
  • Embedding strategy: create_embeddings() flattens child chunks only, batches them, creates embeddings, and writes vectors with the same metadata key used for parent reconstruction.
  • Main execution order: fetch documents -> create chunks -> assign parent ids -> initialize and save SQLite -> embed child chunks -> print "Ingestion complete".
6. answer.py implementation map - Implementation-complete view
  • Runtime config: MODEL = "openrouter/openai/gpt-oss-120b", Chroma path preprocessed_hierarchic_db, SQLite path parent_chunks.db, and a knowledge-base path used for directory-aware retrieval breadth.
  • Answer prompt: SYSTEM_PROMPT requires accurate, relevant, complete answers, bans invention, and explicitly asks for qualifying details on holistic or aggregate questions.
  • Directory-aware utilities: build_directory_doc_counts() counts docs per folder; fetch_parent_chunks_batch() hydrates many parent chunks with a single SQL IN (...) query.
  • Collection access: get_collection() opens or creates the Chroma collection at runtime.
  • Candidate merge: merge_chunks() merges candidate lists and deduplicates by child_chunks[0].
  • Child-first retrieval: fetch_context_unranked() embeds the query, searches child vectors, extracts parent_id, then rebuilds each result with parent context from SQLite.
  • Reranker safeguards: sanitize_rank_order() cleans malformed rank outputs, and rerank() falls back to the original order on failure.
  • Single-doc strategy: fetch_context_single_doc() rewrites the query, searches with both original and rewritten forms, merges, reranks, and returns top FINAL_K=10.
  • Multi-hop strategy: decompose_query() emits 1-5 sub-queries; fetch_context_multi_hop() searches the original question plus every sub-query.
  • Directory-scan strategy: fetch_context_directory_scan() computes adaptive_retrieval_k = doc_count + 5 for holistic or aggregate questions.
  • Tool-calling contract: the tools array defines the JSON schema for all three strategies, including the allowed directory enum company/products/employees/contracts/unknown.
  • Router execution: handle_tool_call() parses JSON arguments and dispatches to Python functions; fetch_context_router() forces tool_choice="required".
  • Generation step: make_rag_messages() builds context from source, parent headline, parent chunk, and the matched child chunk, then adds the original question and optional history.
  • Evaluator schema: RetryEvaluation has only allow_retry and refined_query; the evaluator prompt decides whether retrieval should run again.
  • Retry loop: answer_question() runs retrieve -> generate -> judge -> retry at most once; the retry changes only search_query, while generation still uses the original user question.
  • Reliability wrapper: rewrite, decompose, rerank, route, and answer generation are wrapped in @retry(wait=wait).
Section 6

Workflow / Pipeline - Quy trình / luồng hoạt động

Sơ đồ dưới đây là “bản đồ tư duy” tốt nhất để hiểu hệ thống. Hãy đọc nó theo hai nửa: bên trái là build-time / ingest-time, bên phải là request-time / serving-time.

Diagram reading hint - Cách đọc sơ đồ: dưới lên bên trái là data preparation; giữa sang phải là query routing, answer generation và evaluator retry loop.

Definitions from the diagram - Định nghĩa từ sơ đồ

  • Parent DB: kho lưu parent chunks lớn, giàu ngữ cảnh, được truy hồi bằng parent_id.
  • Child chunk: mẩu văn bản nhỏ 1-2 câu, tối ưu cho semantic matching - khớp ngữ nghĩa chính xác.
  • Vector DB: nơi lưu embedding của child chunks để truy xuất gần đúng theo vector similarity.
  • Query Router Agent: LLM quyết định dùng retrieval tool nào dựa trên loại câu hỏi.
  • Tool Call 1/2/3: ba chiến lược search khác nhau thay vì một pipeline cứng.
  • Self-Reflective Evaluator Agent: bộ kiểm tra nội bộ xem answer đủ tốt chưa, có cần retry không.

What the diagram is really saying

Repo này tách retrieval thành hai tầng: search narrowly, answer broadly. Nghĩa là hệ thống tìm bằng child chunks nhỏ để không bỏ lỡ detail, nhưng trả context bằng parent chunks lớn để LLM không bị thiếu ngữ cảnh. Sau đó, thay vì luôn đi theo một đường thẳng, query được router phân loại rồi đi qua tool phù hợp nhất.

Build-time flow - Luồng xây dữ liệu trước khi chạy
  1. Đọc markdown documents từ knowledge base.
  2. LLM chia mỗi document thành nhiều parent chunks và danh sách child chunks tương ứng.
  3. Gán parent_id cho từng parent chunk.
  4. Lưu parent chunks vào SQLite parent_chunks.db.
  5. Tạo embeddings cho toàn bộ child chunks và lưu vào Chroma preprocessed_hierarchic_db.
Request-time flow - Luồng xử lý khi user đặt câu hỏi
  1. User gửi query.
  2. fetch_context_router() dùng LLM tool calling để chọn: single fact, multi-hop hoặc directory scan.
  3. Tool được chọn sẽ rewrite query hoặc decompose query rồi search vector DB bằng embeddings.
  4. Kết quả vector search trả về child chunks + metadata chứa parent_id.
  5. Hệ thống fetch parent chunks từ SQLite để “nâng” context từ mức chi tiết lên mức hiểu được.
  6. Các chunks được merge, deduplicate và rerank.
  7. LLM sinh answer từ original question + top reranked context.
  8. Evaluator agent kiểm tra chất lượng answer. Nếu thấy thiếu, nó tạo refined query và chạy lại thêm một vòng.
Exact build-time walkthrough - Implementation-accurate ingest flow
  1. fetch_documents() đọc toàn bộ markdown và giữ nguyên ba trường type, source, text.
  2. make_prompt() ước lượng số parent chunks từ độ dài tài liệu rồi ép model tuân thủ overlap, full coverage, child-first retrieval design và explicit entity rewriting.
  3. process_document() dùng structured output để chunk hierarchy được validate bởi Pydantic ngay tại thời điểm ingest.
  4. create_chunks() chạy chunking song song bằng multiprocessing.Pool.
  5. assign_parent_ids() gán join key để child hits sau này nối lại đúng parent context.
  6. insert_parent_chunk() chỉ lưu parent chunks vào SQLite.
  7. create_embeddings() chỉ embed child chunks, batch request và lưu vectors cùng metadata vào Chroma.
  8. Kết quả cuối là kiến trúc dual-store: SQLite cho broad context, Chroma cho precise semantic matching.
Exact request-time walkthrough - Implementation-accurate answer flow
  1. answer_question() bắt đầu với search_query = question và tối đa hai attempts.
  2. fetch_context_router() bắt buộc router chọn đúng một retrieval tool trước khi generate answer.
  3. Strategy được chọn có thể rewrite query, decompose thành sub-queries hoặc tăng retrieval breadth theo directory size.
  4. fetch_context_unranked() search child vectors, lấy parent_id và hydrate parent chunks từ SQLite.
  5. merge_chunks() và rerank() tạo ra final ranked context set.
  6. make_rag_messages() luôn generate bằng original question, kể cả khi retrieval retry dùng refined query.
  7. evaluate_answer_quality() quyết định answer đầu tiên có thiếu tới mức cần thêm một vòng retrieve/generate hay không.
  8. Nếu retry được bật, code chỉ đổi search_query; intent của answer vẫn bám theo câu hỏi ban đầu của user.
Why this beats Day 5 - Vì sao sơ đồ này mạnh hơn Day 5

Day 5 chủ yếu cải thiện một pipeline retrieval chung. Sơ đồ này thêm tầng ra quyết định: direct fact không cần quét rộng như holistic; spanning không thể chỉ rewrite một query là đủ; câu holistic cần retrieval breadth lớn hơn. Tức là hệ thống bắt đầu tối ưu chiến lược, không chỉ tối ưu câu truy vấn.

Section 7

Techniques - Kỹ thuật sử dụng

Technique Problem - Vấn đề How it works - Cách chạy Trade-off - Đánh đổi
Hierarchical semantic chunking Chunk nhỏ dễ mất ngữ cảnh; chunk lớn khó search chính xác. Tạo parent chunk để giữ nghĩa lớn, child chunk để index chính xác. Ingestion phức tạp và tốn token hơn recursive splitter.
Query rewriting - viết lại truy vấn User query mơ hồ hoặc thiếu entity. LLM tạo refined query ngắn và searchable hơn. Có nguy cơ query dilution - làm mất chi tiết quan trọng.
Query decomposition - phân rã truy vấn Spanning question cần nhiều facts ở nhiều nơi. Chia câu hỏi thành sub-queries độc lập để search nhiều hướng. Recall tăng nhưng noise và chi phí cũng tăng.
Adaptive retrieval K Câu holistic cần breadth lớn hơn direct fact. Tính K theo số tài liệu trong thư mục mục tiêu. Nếu K quá lớn, prompt có thể noisy.
LLM reranking - tái xếp hạng bằng LLM Context đúng có trong candidate set nhưng xếp thấp. LLM đọc danh sách chunks và trả về thứ tự relevance. Nếu candidate set sai từ đầu, reranker không cứu được.
Self-reflective retry Answer đầu tiên thiếu hoặc vague. Evaluator judge answer rồi tạo refined query để retry một lần. Tăng latency; cần chặn retry vô hạn.

Implementation Insights - Góc nhìn triển khai

implementation/ingest.py

  • fetch_documents(): đọc knowledge base thành danh sách document.
  • Chunk schema: định nghĩa parent headline, parent chunk, child chunks.
  • process_document(): dùng LLM để semantic chunking có cấu trúc.
  • assign_parent_ids(): tạo khóa nối giữa child retrieval và parent context.
  • create_embeddings(): chỉ embed child chunks và lưu vào Chroma.

implementation/answer.py

  • fetch_context_unranked(): vector search trên child chunks rồi tra lại parent DB.
  • rerank(): LLM reranker sắp lại candidate chunks.
  • rewrite_query() và decompose_query(): tối ưu truy vấn theo từng loại câu hỏi.
  • fetch_context_router(): router agent dùng tool calling.
  • evaluate_answer_quality(): quality gate trước khi trả answer cuối.

  • init_parent_chunk_db(): ensures the parent SQLite store exists before writing.
  • insert_parent_chunk(): persists parent chunks only after parent_id assignment.

  • Chunk.as_result(): converts structured chunk output into the shared runtime object shape.

  • fetch_parent_chunks_batch(): reduces I/O by hydrating many parent ids in one SQL query.
  • merge_chunks(): merges candidate sets from multiple query forms or sub-queries.
  • fetch_context_unranked(): performs the core child-vector search then parent reconstruction.
  • sanitize_rank_order(): protects against malformed reranker output.

  • rewrite_query(): optimizes direct lookup questions.

Detailed ingest.py walkthrough - Giải thích chi tiết theo nhóm chức năng

File implementation/ingest.py là toàn bộ pipeline chuẩn bị dữ liệu cho Agentic RAG. Vai trò của nó không chỉ là "chia chunk và embed", mà là thiết kế lại knowledge base thành một cấu trúc hai lớp: lớp context dùng để trả lời và lớp index dùng để tìm kiếm chính xác.

  1. 1. Initialization and storage setup: file khởi tạo model chunking, embedding model, retry policy, OpenAI client, Chroma path và SQLite path. Ngay từ đây có thể thấy ingest được thiết kế cho dual-store architecture chứ không phải một vector DB phẳng.
  2. 2. Parent store creation: init_parent_chunk_db() tạo bảng parent_chunks. Đây là nơi lưu các đoạn văn bản lớn, đủ ngữ cảnh, để runtime có thể "nâng" từ child hit lên parent context khi cần trả lời.
  3. 3. Raw document normalization: fetch_documents() duyệt knowledge base theo thư mục con, đọc từng file markdown và chuẩn hóa tất cả về cùng một shape dữ liệu. Việc giữ lại type và source ngay từ đầu giúp metadata nhất quán xuyên suốt pipeline.
  4. 4. Structured output contracts: Result, Chunk và Chunks là lớp bảo vệ quan trọng. Thay vì tin rằng model sẽ luôn trả text đúng format, pipeline dùng Pydantic để ép đầu ra vào đúng schema ngay lúc chunking.
  5. 5. Chunking prompt as architecture control: make_prompt() chính là nơi phần lớn tư duy kiến trúc được encode vào ingest. Prompt không chỉ nói "hãy chia tài liệu", mà nói rõ parent chunk dùng cho contextual grounding, child chunk dùng cho precise retrieval, yêu cầu overlap 10-20%, phủ toàn bộ tài liệu, child chunk dài 1-2 câu và không được để unresolved pronouns.
  6. 6. LLM chunk generation: process_document() là đơn vị xử lý cốt lõi cho một tài liệu. Nó gọi LiteLLM với response_format=Chunks, parse kết quả vào schema, rồi convert sang danh sách Result để dùng thống nhất ở các bước sau.
  7. 7. Parallel ingestion: create_chunks() dùng multiprocessing.Pool để xử lý nhiều document song song. Điều này cho thấy ingest được tối ưu cho throughput, nhưng code cũng thừa nhận trade-off bằng comment: nếu rate limit thì hạ WORKERS xuống 1.
  8. 8. Join-key assignment: assign_parent_ids() gán parent_id sau khi chunking hoàn tất. Đây là khóa liên kết quan trọng nhất giữa child vectors và parent context. Không có bước này thì retrieval child-first sẽ không hydrate lại được ngữ cảnh lớn.
  9. 9. Parent persistence: insert_parent_chunk() lưu parent chunks vào SQLite bằng INSERT OR REPLACE. Metadata được serialize thành JSON để runtime có thể tái tạo thông tin nguồn khi cần.
  10. 10. Child embedding pipeline: create_embeddings() flatten toàn bộ child chunks, chia batch, tạo embeddings rồi lưu vào Chroma cùng metadata. Chỉ child chunks được embed, vì child layer được tối ưu cho semantic matching còn parent layer được tối ưu cho answer grounding.
  11. 11. End-to-end execution order: phần __main__ cho thấy thứ tự đầy đủ của ingest: load documents -> create chunks -> assign parent ids -> save parent store -> build child vector store. Điều này phản ánh rõ đây là một data engineering pipeline hoàn chỉnh chứ không phải script tiện ích nhỏ.
Detailed answer.py walkthrough - Giải thích chi tiết theo nhóm chức năng

File implementation/answer.py là runtime chính của toàn hệ thống Agentic RAG. Thay vì dùng một retrieval flow cố định cho mọi query, file này tổ chức một chuỗi quyết định gồm: chọn strategy, retrieve theo strategy đó, tái xếp hạng context, generate answer, rồi tự đánh giá để retry nếu cần.

  1. 1. Runtime initialization and answer contract: phần đầu file thiết lập model runtime, kết nối tới Chroma và SQLite, cùng với SYSTEM_PROMPT quy định chuẩn đầu ra cho answer. Prompt này rất quan trọng vì nó ép model không chỉ trả lời đúng mà còn đủ, nhất là với câu hỏi aggregate hoặc holistic.
  2. 2. Retrieval-oriented data contract: schema Result ở runtime không giống chunk schema của ingest. Nó đại diện cho một context item đã được tái hợp nhất, gồm parent headline, parent chunk, child hit và metadata. Đây là object mà các bước rerank và generation thật sự tiêu thụ.
  3. 3. Storage-aware helper layer: build_directory_doc_counts(), fetch_parent_chunks_batch() và get_collection() là các hàm hạ tầng giúp runtime dùng hiệu quả dual-store design. Đặc biệt, fetch_parent_chunks_batch() cho thấy tác giả chủ động tối ưu I/O bằng batch SQL thay vì query từng parent một.
  4. 4. Raw retrieval reconstruction: fetch_context_unranked() là heart của retrieval. Nó embed query, search trên child vectors, lấy parent_id từ metadata, rồi query SQLite để lấy parent chunks tương ứng. Như vậy retrieval thực chất là quy trình hai bước: tìm nhỏ trước, kéo ngữ cảnh rộng sau.
  5. 5. Candidate post-processing: sau retrieval thô, hệ thống dùng merge_chunks() để hợp nhất candidate sets từ nhiều query forms, rồi dùng rerank() để sắp lại theo relevance thực sự với câu hỏi gốc. sanitize_rank_order() là lớp phòng vệ giúp hệ thống sống sót khi reranker trả về JSON chưa sạch.
  6. 6. Query optimization primitives: rewrite_query() và decompose_query() là hai primitive khác nhau. Rewrite phù hợp cho direct lookup, còn decomposition phù hợp cho spanning hoặc multi-hop queries nơi một query duy nhất khó kéo đủ dữ liệu.
  7. 7. Three concrete retrieval tools: runtime triển khai ba chiến lược thật sự: fetch_context_single_doc() cho direct fact, fetch_context_multi_hop() cho spanning, và fetch_context_directory_scan() cho holistic hoặc aggregate. Khác biệt chính giữa chúng là cách tạo query set và breadth của retrieval.
  8. 8. Router as decision layer: mảng tools mô tả contract function-calling, còn fetch_context_router() đóng vai trò agent định tuyến. Nó không trả lời user mà chỉ buộc model phải chọn đúng một retrieval strategy với tool_choice="required".
  9. 9. Generation assembly: make_rag_messages() ghép source, parent headline, parent chunk, child chunk, history và original question thành prompt cuối cho answer generation. Việc giữ original question ở đây rất quan trọng để refined query không làm lệch ý định thật của người dùng.
  10. 10. Runtime evaluation and self-correction: RetryEvaluation và evaluate_answer_quality() biến offline evaluation mindset thành một control loop trong runtime. Judge không tạo câu trả lời mới; nó chỉ quyết định answer hiện tại có đủ tốt không và nếu chưa thì refined query nào sẽ giúp retrieval tốt hơn.
  11. 11. Main orchestration loop: answer_question() là điểm hợp nhất toàn bộ hệ thống. Flow thực tế là retrieve by router -> generate answer -> judge answer -> nếu cần thì retry thêm đúng một vòng với refined query. Đây là nơi hệ thống thực sự trở thành "agentic", vì nó có khả năng quan sát kết quả trung gian và tự đổi chiến lược truy xuất.
  12. 12. Reliability strategy: nhiều hàm quan trọng được bọc bằng @retry(wait=wait). Điều này cho thấy runtime được thiết kế để chịu được lỗi tạm thời từ API, chứ không assume mọi lời gọi model đều thành công ngay lần đầu.

Section 10

Pitfalls - Lỗi / bẫy thường gặp

Failure modes - Trường hợp dễ fail

  • Router misclassification - router chọn nhầm đường: câu spanning bị xem như direct fact làm hệ thống search quá hẹp.
  • Weak decomposition - phân rã truy vấn kém: sub-queries quá giống nhau hoặc bỏ sót mảnh thông tin then chốt.
  • Query dilution - loãng truy vấn: refined query bỏ mất số liệu, entity, điều kiện thời gian.
  • Child-parent mismatch: child chunk match đúng nhưng parent chunk quá rộng hoặc chứa nhiều nhiễu.
  • Retry overuse: evaluator cứ thấy answer chưa đẹp là retry, làm chi phí và latency tăng nhanh.

Common misunderstandings - Hiểu lầm phổ biến

  • “Có agent là thông minh hơn”: sai; agent chỉ đáng dùng nếu nó map đúng vào một decision point có giá trị.
  • “Rerank rồi thì không cần recall mạnh”: sai; reranker không thể cứu context chưa từng được retrieve.
  • “Chunk càng nhỏ càng tốt”: sai; chunk nhỏ tốt cho search nhưng tệ cho grounding nếu không có parent context.
  • “Eval chỉ để làm dashboard”: sai; repo này cho thấy tư duy eval có thể đi vào runtime để điều khiển retry.

    A: Nó phân loại intent của query và gọi retrieval tool phù hợp thay vì dùng một flow duy nhất.

    A: Vì thông tin cần thiết nằm rải ở nhiều nơi; một query đơn thường không kéo đủ các mảnh liên quan.

    A: Offline eval chỉ đo sau khi build; evaluator loop dùng đánh giá ngay trong runtime để quyết định có retry không.
